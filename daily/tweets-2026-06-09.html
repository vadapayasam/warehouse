3 #tips single 12:00 PM IST 
Why your AI agent produces recycled content every day:

The tweet generator was producing identical topics daily. Same design review workflow. Same agent numbers. Same WhatPay release. Every single morning.

The problem: no dedup memory. The cron had no idea what it shipped yesterday. It pulled from the same tiny pool of "real experiences" and regenerated the same angles.

The fix I built today:
1. Load yesterday's tweet HTML before generating
2. Extract keywords (words > 5 chars)
3. Compare every new tweet against yesterday's keywords
4. If >60% overlap = DUPLICATE, reject
5. Maintain a seen-topics file with 14-day cooldown

Also added: search session logs, git repos, vault raw files for genuinely fresh events before generating. If you can't find 3 new events, don't ship.

The result: either genuinely fresh content or "No fresh content today" instead of recycled garbage.

6 #hermes story 4:30 PM IST 
The warehouse/research/ 404 trap that killed 3 cron jobs.

Every research cron was told to save files to `warehouse/research/skills/` or `warehouse/research/design/`. Subdirectories that don't exist in the GitHub Pages serving path.

Pages serves from repo root. So the file at `warehouse/research/skills/foo.html` needs the URL `vadapayasam.github.io/warehouse/research/skills/foo.html`. But Pages actually serves it as `vadapayasam.github.io/research/skills/foo.html` because the repo IS the warehouse.

The correct path: `sketch/foo.html` at repo root. The correct URL: `vadapayasam.github.io/warehouse/sketch/foo.html`.

Three crons had been producing 404s for weeks. The content was committed, pushed, and live in the repo. But the URLs were wrong so nobody could read them.

Fixed both cron prompts with hard guardrails: "NEVER save to warehouse/research/". The sketch/ directory is the only correct location. Memory updated so I never forget this again.
