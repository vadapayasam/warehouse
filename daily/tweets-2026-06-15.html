Working on LeaveHack this week.

SwiftUI + WidgetKit combo is still undefeated for quick utility apps. Here's why:

Live Activities: Lock screen widget shows remaining leave days. No app open needed. This is the "always visible" feature that makes a utility app stick.

SwiftUI List + Form: Built the entire settings screen in 40 lines. No custom UI. No Auto Layout wrestling. System handles dark mode, dynamic type, accessibility.

App Intents: Siri can add a leave day. "Hey Siri, I'm taking Friday off." Integration took 2 hours. Not because it's complex — because Apple made the API actually usable for once.

WidgetKit timeline: Widget refreshes at midnight automatically. No background fetch. No push notifications. System updates the view when the date changes.

Total build time: 3 evenings. ~8 hours. That's the power of staying inside the system instead of fighting it.

4 #apps single 4:30 PM IST 
My iOS app philosophy in one sentence:

Make the thing you wish existed. Ship in 2 weeks. Iterate in public.

The breakdown:

'Make the thing you wish existed' — WhatPay started because I couldn't find a simple Indian expense tracker that didn't want my email and phone number. TimeLeft started because I wanted a countdown widget that looked good.

'Ship in 2 weeks' — Not because I'm fast. Because I scope aggressively. v1.0 has one core flow. Everything else is v1.1. The first version of WhatPay tracked expenses and showed a total. That's it. No categories, no export, no iCloud.

'Iterate in public' — Build numbers are public. TestFlight links go to friends. I post screenshots before the app is done. The feedback shapes the product more than any user research session.

The counter-intuitive part: shipping unfinished feels risky. But the real risk is building something perfect that nobody wants.
5 #failures story 2:00 PM IST 
My Honcho memory system hit capacity. Context windows were getting compressed. Facts were being dropped.

The symptoms:
- Agent started repeating questions I had answered before
- Cross-references in the wiki were getting stale
- The daily digest was missing items that should have been tracked

I checked the memory store: 2,147 entries. The system was deduplicating but the dedup algorithm was too conservative.

The fix was "goblin surgery" — my term for aggressive manual cleanup:

1. Deduplicated entries: 200+ near-duplicate facts merged
2. Pruned stale facts: removed 180 entries older than 6 months with no recent references
3. Reorganized structure: split the monolithic memory file into category-specific files (ios, design, workflow, failures, tools)
4. Added a TTL policy: facts auto-expire after 12 months unless renewed by reference

Result: 1,400 entries. Faster lookups. Better context retention.

The lesson: persistent memory needs maintenance. Left alone, it becomes a hoarder's attic.

Sometimes you need to delete to grow.
