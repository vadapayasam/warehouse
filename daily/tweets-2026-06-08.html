How a single checkbox cost me 2 hours:

Google OAuth broke on my Hermes setup. Every Docs upload failed with 403 access_denied. The error message was useless: just "insufficient permissions" with a link to a dead troubleshooting page.

I went through the entire OAuth flow twice. Regenerated credentials. Checked scopes. Verified the refresh token. Even reinstalled the Google client library. Nothing.

Turned out I needed to add my email as a "test user" in Google Cloud Console. Not a permission issue. Not a scope issue. Just: my own app didn't recognize me as a valid tester while it was in unpublished mode.

One checkbox. 2 hours of debugging. 4 credential regenerations. 1 very angry afternoon.

Google auth is the YAML of authentication: looks simple, hides complexity, and the error messages actively mislead you.

The real fix wasn't technical. It was reading the fine print on a settings page I skipped because "I'm the owner, why would I need to test my own app?"

🕓 Post at 3:00 PM IST

WhatPay v1.3.2 just shipped build 12. Here's what 9 App Store builds taught me:

Build 1-3: "Processing" in App Store Connect means nothing. It can say "Processing" for 20 minutes or 2 hours. The actual state is opaque. I now ignore it and check TestFlight directly after 15 minutes.

Build 4: The stuck modal bug. URL scheme handler triggered unconditionally on launch. Every screenshot showed the same system dialog. Took 2 days to trace it to a deep link handler that fired before the app finished launching. Fixed by gating the handler behind a launch completion flag.

Build 5-6: Privacy manifest anxiety. Apple started requiring privacy manifests in early 2024. WhatPay doesn't track users or use third-party SDKs, but the manifest still needs to exist and be correct. One wrong key and the submission gets rejected with a generic error.

Build 7: The CSV export lie. v1.0 on the App Store claimed "Export data as CSV." It didn't. v1.1 removed the claim and correctly advertises iCloud sync. Building honest copy is harder than building features.

Build 8: First successful asc CLI upload. No Xcode Organizer. No manual IPA export. Just: archive, export with API key auth, asc builds upload. 30 seconds. The moment I stopped fearing the submission pipeline.

Build 9-11: Settings crown badge experiment. Tried a "Pro" crown badge on the settings row. Non-Pro users see it. Pro users see a checkmark. The banner padding was wrong on iPhone 14 Pro: the X button got buried. Fixed by pushing the banner down 24pt.

Build 12: Clean. Merged feature branch to main. Final banner padding fix. Ready for review.

The thing nobody tells you about App Store submission: every build teaches you something the documentation doesn't cover. The docs are correct. They just don't tell you what actually happens.

LeaveHack: SwiftUI + WidgetKit is still undefeated for quick utility apps.

Building LeaveHack this week. It's a simple leave/PTO tracker. The kind of app you build in a weekend and ship to TestFlight on Monday.

Why SwiftUI + WidgetKit wins for this:

Live Activities: The widget shows remaining leave days on your lock screen. No app open required. This is the "always visible" feature that makes a utility app stick.

SwiftUI List + Form: Built the entire settings screen in 40 lines. No custom UI. No Auto Layout wrestling. The system handles dark mode, dynamic type, and accessibility out of the box.

App Intents: Siri can add a leave day. "Hey Siri, I'm taking Friday off." The integration took 2 hours. Not because it's complex: because Apple made the API actually usable for once.

WidgetKit timeline: The widget refreshes at midnight automatically. No background fetch. No push notifications. The system just updates the view when the date changes.

The total build time so far: 3 evenings. Maybe 8 hours total. That's the power of staying inside the system instead of fighting it.

The counterpoint: if I wanted a custom animation or a non-standard layout, SwiftUI would fight me. But for a utility app? Stay boring. Ship fast.

🕕 Post at 6:00 PM IST

For iOS devs building alone: stop using Xcode Organizer.

Set up asc (App Store Connect CLI) once. Then never touch Xcode Organizer again.

The pipeline I use for every WhatPay build:

1. xcodebuild archive -scheme WhatPay -destination 'generic/platform=iOS'
2. xcodebuild -exportArchive -exportOptionsPlist export.plist (uses app-store-connect method + API key auth)
3. asc builds upload --ipa WhatPay.ipa

Total time: 30 seconds. The upload happens in the terminal. No web interface. No drag-and-drop. No "Processing" anxiety.

The setup cost: generating an API key in App Store Connect, saving it locally, creating an export options plist. Took me 20 minutes the first time. Saves me 5 minutes per build forever.

The catch: you need a proper provisioning profile and signing certificate. The CLI doesn't help with setup: it assumes your signing is correct. If your archive fails, fix signing first.

The second catch: if no Apple ID is signed into Xcode, export fails with a cryptic error about "no account found." This happened to me on June 7. The fix: sign into Xcode > Preferences > Accounts, or use manual signing with a downloaded provisioning profile.

The third catch: asc is version 1.2.7. It's not perfect. Some commands are undocumented. But for builds upload and builds list, it's rock solid.

What I don't miss: the Xcode Organizer window, the web upload form, the "Your app is Processing" state that means nothing, the 3-minute wait for the upload progress bar.

What I do miss: nothing. Use the CLI.

8 #workflow step-by-step
🕖 Post at 7:00 PM IST

Every design review I do now follows this pipeline. It took 3 iterations to get right:

Iteration 1: Screenshot + Figma comments. Problems: Figma loads slowly on mobile. Comments get lost in threads. Clients reply in email instead of Figma. The feedback is scattered across 3 platforms.

Iteration 2: Screenshot + Google Doc. Problems: Docs don't handle images well at scale. 12 screenshots in a Doc = scroll hell. Comments are better than Figma but still buried. And the Doc link feels temporary.

Iteration 3 (current): Agent captures screenshots → HTML review gallery generated automatically → I add commentary inline → Gallery pushed to GitHub Pages → Link shared. Done.

The pipeline:

1. Agent captures screenshots from the iOS simulator or device. 12 screens for a typical app review. Named sequentially: 01-home, 02-settings, etc.

2. HTML review gallery generated automatically. The template includes: screenshot, UX observation, severity tag (cosmetic / functional / critical), and a notes field. All styled with the same CSS I use for research artifacts. Consistent look.

3. I add commentary inline. Not in a separate tool. Directly in the HTML. I write observations as I scroll through the gallery. The commentary stays with the screenshot it describes. No context switching.

4. Gallery pushed to GitHub Pages. The URL is permanent. I can share it in Slack, email, or a ticket. The recipient sees exactly what I see. No "download this file" or "sign into Figma."

5. Link shared. The recipient opens the URL. Sees the gallery. Reads the observations. Replies with feedback. I update the HTML and push again. The URL stays the same. The content updates.

What I removed: Figma comments, email threads, PDF exports, screenshot zip files, "can you see the attached image?" messages.

What I kept: The actual review. Just faster.
