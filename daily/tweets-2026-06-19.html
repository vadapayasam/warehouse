The last card in LeaveHack v2's stack would freeze. Swipe it and nothing happened. The screen sat there with a half-swiped card, no way to reset it.

I blamed the animation. Then the gesture recognizer. Then SwiftUI's rendering pipeline. Eight hours of blame-shifting before I looked at the actual state variable.

Three commits. Same message. Same minute. "synthesis: daily append 19 Jun 2026."

I know what happened. Tina ran the vault pipeline, produced HTML, pushed to GitHub, and wasn't sure the push worked. So it ran again. And again.

This is a real problem with autonomous agents: they don't know when to stop. The work is done, the push succeeded, but there's no built-in "that's enough" signal. So they retry. Because retrying feels safer than trusting the output.

Git log as a trust metric. If your AI agent made three identical commits with the same message, it doesn't trust itself.

I cleaned it up with a `git rebase`. The three commits became one. But the pattern repeats until the agent has a confirmation signal it actually believes.

3 #tips 4:00 PM IST 
z-index card stacks look right on paper. They fail the moment two cards need to animate at the same time.

My LeaveHack v2 stack was built with card 0 on top, card 1 behind it, card 2 deeper. Simple. Wrong.

When card 0 swipes out and card 1 slides in, both animate simultaneously. z-index can't handle that without visual flicker because the stacking order has to rearrange mid-transition.

Replaced with an indexed array approach: `cards[0]` is always the front card. When dismissed, `cards[1]` becomes `cards[0]` instantly. No z-index fighting, no flicker. The array swap is atomic.

The z-index approach felt cleaner in code. Looked terrible in the simulator. If you're building animated card stacks, test overlapping transitions before settling on a stacking strategy. z-index looks right on paper. Fails on screen.
