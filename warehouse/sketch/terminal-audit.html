Pattern 1: WasteFile reading via cat/head/ls

Every time I type terminal("cat file") or terminal("ls -la"), I pay for the shell overhead AND the full output. read_file and search_files are zero-overhead structured tools.

~1,030 calls / 15% of terminal

Bad (current)

terminal("cat ~/Projects/repo/file.swift")           # ~200 tokens of tool schema + full file output
terminal("head -50 ~/Projects/repo/file.swift")       # slightly better but still shell overhead
terminal("ls -la ~/Projects/repo/")                    # should be search_files(target='files')

Good (optimized)

read_file("~/Projects/repo/file.swift", limit=50)    # Zero terminal schema tokens. Paginated.
search_files(pattern="*.swift", path="~/Projects/repo/")  # 0 schema tokens. Sorted by mtime.

Savings

Each terminal call avoided saves ~25 tokens of tool definition overhead + ~50-200 tokens of output framing. Over 1,000 calls: ~75K-225K tokens saved per month.

Pattern 2: WasteShell grep/find instead of search_files

Same problem as Pattern 1. grep -r, find . -name, and rg all go through terminal, paying tool schema overhead. search_files is ripgrep-backed, native to Hermes, and returns structured results.

~685 calls / 10% of terminal

Bad

terminal("grep -rn 'TODO' ~/Projects/repo/ --include='*.swift'")
terminal("find ~/Projects/repo/ -name 'KANBAN.md'")
terminal("rg 'error' ~/.hermes/logs/ --no-heading | tail -20")

Good

search_files(pattern="TODO", path="~/Projects/repo/", file_glob="*.swift")
search_files(pattern="KANBAN.md", target="files", path="~/Projects/repo/")
search_files(pattern="error", path="~/.hermes/logs/", limit=20)

Additional win: search_files is line-numbered by default

No need to pipe to wc -l. Use output_mode="count" for quick counts, output_mode="files_only" for listing.

Pattern 3: OptimizationBatch git operations into one call

Git is our biggest terminal consumer. The vault-sync cron alone runs daily with 2 calls. Fix: combine porcelain check + conditional chain into a single bash -c block.

~2,050 calls / 30% → target: ~700

Bad : 3 separate terminal calls for one sync

terminal("cd ~/repo && git status --porcelain")       # Tool call #1
terminal("cd ~/repo && git add -A && git commit -m 'x'")  # Tool call #2
terminal("cd ~/repo && git push")                      # Tool call #3

Good : 1 bash -c with conditional

terminal("""
  cd ~/repo || exit 1
  STATUS=$(git status --porcelain)
  if [ -z "$STATUS" ]; then echo '[SILENT]'; exit 0; fi
  git add -A && git commit -m "auto sync" && git push
  echo '[SILENT]'
""")

Better : detect changes first, only then proceed

terminal("cd ~/repo && git status --porcelain | wc -l")  # Quick check: returns "0" or "5"
# If > 0, then single batch call for add+commit+push

Savings

Reducing 3 calls → 2 (quick check + batch) saves ~33% of git terminal calls. If each vault-sync cron iteration saves 1 call × 365 days × all git-using crons = ~1,350 terminal calls saved per year.

Pattern 4: OptimizationUse execute_code for multi-step logic

When you need to read files → process → write back, terminal("python3 -c \"...\"") works but execute_code is cleaner : it avoids escaping hell, supports from hermes_tools import read_file, terminal, and is purpose-built for multi-step Python.

Bad : python3 -c with escaped quotes

terminal("python3 -c \"import json; data = json.load(open('file.json')); print(len(data))\"")

Good : execute_code with proper Python

execute_code(code="""
from hermes_tools import read_file, write_file
import json

data = json.loads(read_file("file.json")['content'])
print(f"Loaded {len(data)} items")
""")

Note: execute_code is blocked in cron mode

For cron jobs, use terminal("python3 -c ...") with single-quote heredocs instead of double-quote escaping:

terminal("python3 -c 'import json; print(len(json.load(open(\"file.json\"))))'")

Or better: write a script to ~/.hermes/scripts/ and reference it via terminal("python3 ~/.hermes/scripts/script.py").

Pattern 5: Quick WinsTruncate output before it hits context

Every terminal call returns its full stdout. A 500-line git log or 200-line test output eats your context window. Always truncate.

Bad

terminal("npm test")       # Returns 300 lines. You read 10.
terminal("git log --oneline --all")  # Returns 150 commits. You need 3.

Good

terminal("npm test 2>&1 | tail -10")                    # Last 10 lines only
terminal("npm test && echo 'PASS' || echo 'FAIL'")       # Just pass/fail
terminal("git log --oneline -5")                         # Last 5 commits
terminal("git log --oneline -3 --since='7 days ago'")    # Last week only

Proposed: tm git-sync

#!/bin/bash
# ~/.hermes/scripts/tm/git-sync.sh
# Usage: tm git-sync [repo-path] [commit-message]
# Default: ~/.hermes/obsidian-vault, "auto sync $(date +%H:%M)"

REPO="${1:-$HOME/.hermes/obsidian-vault}"
MSG="${2:-auto sync $(date +%Y-%m-%d %H:%M)}"

cd "$REPO" || { echo "FAIL: cd $REPO"; exit 1; }

STATUS=$(git status --porcelain)
if [ -z "$STATUS" ]; then
  echo "[SILENT]"
  exit 0
fi

git add -A && git commit -m "$MSG" && git push 2>&1
echo "SYNCED: $(echo "$STATUS" | wc -l) files"
