File reading waste — cat, head, ls

Every terminal("cat file") pays shell overhead AND full output. read_file and search_files are zero-overhead structured tools.

~1,030 calls/month · 15% of terminal · 90% avoidable Waste

Bad (current)

terminal("cat ~/Projects/file.swift")
terminal("head -50 ~/Projects/file.swift")
terminal("ls -la ~/Projects/")

Good (optimized)

read_file("file.swift", limit=50)
search_files("*.swift", target="files")

Savings: Each avoided call saves ~25 tokens tool schema + ~50-200 tokens output framing. Over 1K calls: ~75K-225K tokens/month.

Shell grep/find instead of search_files

grep -r, find ., and rg all go through terminal, paying tool schema overhead. search_files is ripgrep-backed and returns structured results.

~685 calls/month · 10% of terminal · 90% avoidable Waste

Bad

grep -rn 'TODO' ~/repo/ --include='*.swift'
find ~/repo/ -name 'KANBAN.md'
rg 'error' ~/.hermes/logs/ | tail -20

Good

search_files("TODO", file_glob="*.swift")
search_files("KANBAN.md", target="files")
search_files("error", limit=20)

Bonus: search_files is line-numbered by default. Use output_mode="count" for quick counts, output_mode="files_only" for listing.

Batch git operations into one call

Git is the biggest terminal consumer. The vault-sync cron alone runs daily with 2-3 calls. Fix: combine porcelain check + conditional chain.

~2,050 calls/month · 30% of terminal · target: ~700 Partial

Bad — 3 calls

# Call 1: status check
terminal("git status --porcelain")

# Call 2: add + commit
terminal("git add -A && git commit -m 'x'")

# Call 3: push
terminal("git push")

Good — 1 bash -c

terminal("""
  cd ~/repo || exit 1
  STATUS=$(git status --porcelain)
  if [ -z "$STATUS" ]; then
    echo '[SILENT]'; exit 0
  fi
  git add -A && git commit -m "sync" && git push
  echo '[SILENT]'
""")

Savings: 3 calls → 1. If each cron saves 1 call × 365 days × all git crons = ~1,350 terminal calls/year saved.

Use execute_code for multi-step logic

terminal("python3 -c \"...\"") works but execute_code avoids escaping hell and supports from hermes_tools import read_file.

Bad — escaped quotes

terminal("python3 -c \"import json; \\
  data = json.load(open('f.json')); \\
  print(len(data))\"")

Good — clean Python

execute_code(code="""
from hermes_tools import read_file
import json
data = json.loads(
  read_file("f.json")['content']
)
print(f"Loaded {len(data)} items")
""")

Note: execute_code is blocked in cron mode. For cron, write a script to ~/.hermes/scripts/ and call via terminal("python3 ~/.hermes/scripts/script.py").

Truncate output before it hits context

Every terminal call returns full stdout. A 500-line git log eats your context window. Always truncate.

Instead ofUseSaves
git loggit log -5 / --since=...~200+ lines
docker psdocker ps --format '{{.Names}}'~50 chars/container
ls -lals -1 | head -20~80% reduction
findfind ... | head -10~90% reduction
cat fileread_file(path, limit=50)Zero terminal overhead
npm testnpm test | tail -10~290 lines

Quick rule: Always pipe build/test output through | tail -20 or check exit code only: cmd && echo PASS || echo FAIL.

Pre-flight checklist for every terminal call

Before hitting enter on any terminal(...), ask:

QuestionIf yesDefault to
Am I reading a file? Use read_file read_file(path, limit=50)
Am I searching content? Use search_files search_files(pattern, file_glob)
Am I finding files? Use search_files search_files(pattern, target='files')
Am I editing a file? Use patch patch(path, old, new)
Git status + add + commit + push? Use tm git-sync tm git-sync ~/repo "msg"
Deploying HTML? Use tm deploy-html tm deploy-html file.html
Multi-step Python logic? Use execute_code execute_code(code="""...""")
Output going to be long? Pipe through tail cmd 2>&1 | tail -20
Build or test? Check exit code only cmd && echo PASS || echo FAIL

System rule (proposed)

# Before every terminal() call, ask:
"Can I do this with read_file, search_files, patch,

If yes, use the dedicated tool. Terminal is for builds,
git pushes, Docker, and commands with side effects.
