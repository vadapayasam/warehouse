Dynatrace Davis AI — Agentic AI Assessment

Research week: June 8, 2026 | Issue #6

Focus: Workflow lifecycle, automation boundaries, where a request starts and ends

TL;DR: Dynatrace has the most mature observability-native AI engine in the market. Davis AI correlates anomalies across topology, performs causal root-cause analysis, and now drives autonomous remediation through Workflows and the MCP Server. For enterprise SRE teams, this is best-in-class. For MSPs, it is a powerful APM layer that requires custom bridging to PSA tools, client separation, and billing alignment.

1. Product Overview

Dynatrace is a unified observability platform founded in 2005 (HQ: Waltham, MA), publicly traded since 2019. Its AI engine, Davis AI, has been the platform's core differentiator for over a decade. In early 2026, Dynatrace rebranded its AI suite as Dynatrace Intelligence — an "agentic operations system" that fuses deterministic causal AI with generative capabilities.

The 2026 platform consists of three layers:

  • Grail — unified data lakehouse (logs, metrics, traces, events, security findings)
  • Smartscape — real-time auto-discovered topology and dependency graph
  • Dynatrace Intelligence — reasoning layer with causal AI, predictive forecasting, and generative AI (Davis CoPilot / Assist)

Key 2026 launches:

  • Dynatrace Assist (June 2026) — evolution of Davis CoPilot; conversational interface with multi-step reasoning
  • Dynatrace MCP Server (Feb 2026, GA) — open Model Context Protocol bridge to external AI agents
  • Dynatrace Intelligence Agents — ready-made domain agents for specific use cases
  • Agentic Workflows — event-triggered or scheduled automations with embedded Davis CoPilot

2. Workflow & Lifecycle Analysis

Incident Lifecycle: From Signal to Resolution

Dynatrace maps a precise, deterministic lifecycle for every incident:

[Signal Detection]
    |
    v
[Davis Event Created]  <-- anomaly detected (threshold breach, baseline degradation, crash)
    |
    v
[Processing State]       <-- up to 3 min: causal AI correlates events, avoids duplicates
    |
    v
[Problem Opened]         <-- grouped, correlated problem with root cause + impact analysis
    |

    |
    +--> [Agentic Workflow Trigger]  --> auto-remediation, ticket creation, resource scaling
    |
    +--> [Human Investigation]       --> Dynatrace Assist, drill-down to code level
    |

    |
    v
[Reopening Period]       <-- 30 min window: new events can reopen the problem
    |
    v

Automation Boundaries

Dynatrace Workflows supports three automation modes:

  • Notification/orchestration — fully automated: post to Slack, create Jira ticket, send email
  • Guided remediation — AI suggests steps, human approves: Dynatrace Assist generates remediation guidance, human triggers action
  • Autonomous action — fully automated with guardrails: resource scaling, manifest edits, runbook execution (requires high-confidence Davis AI forecasting + explicit workflow configuration)
Guardrail: Autonomous Workflows require explicit enablement and are typically scoped to low-risk, high-confidence actions (e.g., scaling a Kubernetes deployment based on forecasted demand). There is no general "auto-fix everything" mode.

Dynatrace Assist (June 2026)

Dynatrace Assist is the conversational interface to Dynatrace Intelligence. It replaces Davis CoPilot with deeper multi-step reasoning:

  • Pulls context from Grail + Smartscape + active agents in real time
  • Shows transparent reasoning: which data backs each recommendation
  • Handles follow-up drill-downs without breaking conversational flow
  • Can explain raw logs, suggest vulnerability fixes, and guide remediation

5. Pricing Intelligence

Dynatrace uses pure consumption pricing. AI capabilities are bundled into platform tiers; there is no separate "AI add-on" fee.

Pricing insight: Unlike PagerDuty Advance ($415/mo flat) or Atlassian Rovo ($1/resolution), Dynatrace AI is "free" once you pay for observability data. The cost scales with infrastructure size, not with AI usage. For a 40-host environment: ~$2,320/mo for Full-Stack = includes full AI. This is competitive for mid-market but becomes expensive at scale compared to flat-rate alternatives.

MSP-Specific Gaps

No native MSP features found. Dynatrace is built for enterprise SRE and DevOps, not multi-tenant MSP operations. The following are missing or require custom work:

  • No PSA integration: No native ConnectWise, Autotask, or HaloPSA connectors. ServiceNow and Jira are supported, but MSP-specific PSA workflows (ticket-to-billing, time entry, client portals) are absent.
  • No client separation: Multi-tenancy requires manual uniform tagging and naming conventions. Per-client baselines are possible but not a first-class feature.
  • No MSP billing alignment: Per-host pricing does not map to per-client or per-ticket billing models. No resolution-based or per-consumption unit that MSPs can pass through.
  • No co-managed templates: No pre-built runbooks or workflows for typical MSP scenarios (patch management approval, endpoint health checks, client onboarding).
  • No cross-vendor AI orchestration: The MCP Server connects external agents TO Dynatrace, but Dynatrace agents cannot natively orchestrate actions in non-Dynatrace tools (e.g., trigger a ConnectWise script or a NinjaOne policy change).

7. Verdict

For enterprise SRE/DevOps: Dynatrace is the most mature observability-native AI platform. The causal AI engine, real-time topology (Smartscape), and the MCP Server make it a genuine "agentic operations system" rather than a bolt-on AI feature. The shift from Davis CoPilot to Dynatrace Assist (June 2026) signals serious investment in generative + deterministic hybrid AI.

For MSPs: Dynatrace is a powerful APM/observability layer that an MSP can resell or use internally, but it is not an MSP-native tool. The gaps are structural: no PSA bridge, no client billing model, no multi-tenancy, no co-managed workflows. An MSP running Dynatrace for clients would need a custom integration layer (webhook router + PSA API mapping + billing normalization) to make the AI insights actionable in their operational stack.

Strategic angle: The MCP Server is the most interesting piece for MSP tooling builders. Because it uses the open Model Context Protocol, a custom MSP agent could theoretically query Dynatrace Intelligence and then act through PSA/RMM APIs. This positions Dynatrace as a data source in a broader agentic ecosystem, rather than a closed loop. For the "universal connector bus" architecture pattern, Dynatrace's MCP Server is a strong inbound signal source.
