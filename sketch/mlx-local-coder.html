Hermes Agent MCP Plugin

Started with a token filter. Ended with a multi-model router.

Three MLX models, one MCP server, zero MB idle. A 3B coder for summarization and codegen. A 1.2B thinking model for gating decisions. Auxiliary vision and TTS models on demand. This is what four months of iteration looks like.

Prabha · Agent Tina · Updated June 2026

Let me kill the fantasy first

None of these models run your agent. They don't understand your codebase. They have 2024 knowledge cutoffs. If you install this expecting a local coding assistant, you will be disappointed.

The job isn't intelligence. The job is filtration, gating, and narrow utility -- reading brew update logs, triaging build errors, checking screenshots, deciding if a cron task should run. The expensive model still does everything that matters.

I started with one model (3B Coder) and three tools (summarize, filter, codegen). It saved 82% on input tokens. That was already worth it.

But over four months, the architecture evolved. I added a thinking model for gating decisions (should this cron job run or skip?). I added a vision model for reading screenshots and UI layouts. A TTS model for voice output over Telegram. All on the same MCP server router, all on-demand subprocess loading, all 0 MB RAM when idle.

What started as a token filter became a multi-model router. This is the full picture.

The Router: one MCP server, five tools, three models

The MCP server at ~/.hermes/scripts/mlx-mcp-server.py is a router. It doesn't run any model directly. It dispatches each tool call to a subprocess that loads the right model, runs inference, produces output, and exits. Every model is a separate subprocess. Every subprocess frees its RAM on exit.

Hermes Agent
  ↓
mlx-coder MCP server (router)
  ¦-- mlx_codegen   → Qwen2.5-Coder-3B   → mlx_lm subprocess
  ¦-- mlx_summarize → Qwen2.5-Coder-3B   → mlx_lm subprocess
  ¦-- mlx_filter    → Qwen2.5-Coder-3B   → mlx_lm subprocess
  ¦-- mlx_think     → LFM2.5-1.2B-Think  → mlx_lm subprocess
  ¦-- mlx_vision    → Qwen2-VL-2B        → mlx-vision.py subprocess
  ¦
  [also registered separately]
  ¦-- mlx_tts       → Qwen3-TTS-0.6B     → mlx-tts-server.py (separate MCP)

The key insight: no model is loaded until a tool is called. The shell wrapper ~/bin/tm provides the same tools for terminal use outside the agent:

$ cat build.log | tm mlx-summarize "errors and warnings"
 - Build completed with 2 warnings (deprecated API usage)
 - Xcode version mismatch warning (15.4 vs 15.3)
 - No errors found

$ tm mlx-think "git status shows 3 modified files. Should I commit?"
<think>3 modified files is a small batch. No large unstaged changes. No untracked binaries. Low risk of mid-task interruption.</think>
Decision: RUN. Small, safe, easy to roll back.

$ tm mlx-vision --prompt "What buttons are on this screen?" /tmp/screenshot.png
The screen shows three buttons: "Save Changes," "Cancel," and "Delete Account."
[vision: 2.25GB RAM peak, 0.9s load, 1.8s gen]

Why two models instead of one? (Coder + Thinker)

The 3B Coder (Qwen2.5-Coder-3B) is good at summarization and codegen. But for decisions -- should this cron task run? Is this anomaly worth flagging? -- it's not designed for chain-of-thought. You get an answer without knowing why.

That's where the thinking model (LFM2.5-1.2B-Thinking) comes in. It produces explicit reasoning before the answer. In cron workflows, this is the difference between:

3B CODER (wrong tool)

Prompt: "Should this cron job run?"

"Yes." No explanation. Can't audit the decision. Same answer for every condition because training data didn't include gating scenarios.

THINKING MODEL (right tool)

Prompt: "Should this cron job run?"

"<think>Last run was 2h ago. Disk at 45%. No ongoing build detected. No errors in recent logs.</think> Decision: RUN." Auditable, debuggable, trustworthy.

The pairing pattern in production:

Two-model gating pattern

LFM 1.2B decides if the task should run -- checks conditions, inspects recent state, produces a structured decision (RUN / SKIP).

3B Coder executes -- only if the thinker says RUN. Does the summarization, filtering, or codegen.

This is used in cron jobs: the thinking model gates execution, the coder model produces output. At 0.9GB and 1.8GB RAM peak respectively, they can share the 16GB M2 comfortably -- just not simultaneously.

The RAM trick: all four models take 0 MB when idle

Every model is loaded on-demand via subprocess. The MCP server itself is a lightweight Python process (~15 MB resident). When a tool is called, the subprocess loads the model from ~2 GB/s disk cache, runs inference, prints the result, and exits. RAM is freed completely.

Most people hear "local model" and imagine a daemon eating 2 GB of RAM 24/7. That's Ollama. That's vLLM. This is not that. The tradeoff is latency (3-10s per call vs <1s on a frontier API), but in agentic workflows -- especially cron jobs -- nobody's waiting interactively.

The vision model: what we went through to find one that works

This was painful. I tried two LFM vision variants first -- both garbage.

Failed: LFM2.5-VL-450M-bf16

861 MB. Config said projector_use_layernorm: false but weights contained layer_norm keys. After patching mlx_vlm to make layer_norm conditional, output was repetitive hallucinated text: "The image is a simple, yet effective, yet not overly complex..." repeated 10+ times. Unusable.

Failed: LFM2.5-VL-1.6B-4bit

1.4 GB. Same layer_norm patch needed. Same garbage output: "There There There..." on one test, empty on others. Root cause suspected: version mismatch with mlx-vlm 0.6.3.

Winner: Qwen2-VL-2B-Instruct-4bit

1.25 GB, 2.25 GB RAM peak, 83 tok/s. Zero configuration. No source patches. First inference produced correct output: "A blue square and a green circle with the text 'Hi' below them." Loads in 0.9s from HF cache. Runs via ~/.hermes/scripts/mlx-vision.py as a subprocess called from the MCP router.

Testing results on a 512x512 test image (blue rect, green circle, text "Hi"):

Model Output RAM Peak Load Gen
Qwen2-VL-2B-4bit (MCP call) Correct: "blue square with text 'Hi', green circle" 2.25 GB 0.9s 1.8s
LFM2.5-VL-450M-bf16 Hallucinated repetition. Unusable. ~1.1 GB ~3s N/A
LFM2.5-VL-1.6B-4bit Garbage output. Unusable. ~1.6 GB ~0.7s N/A

Lesson: When searching MLX vision models, check the conversion version in the HF model card. Qwen2-VL was converted with mlx-vlm 0.0.13 and works flawlessly with 0.6.3. The LFM models were converted with unknown older versions and fail silently with garbled output.

TTS: voice output over Telegram via MLX

Separate MCP server at ~/.hermes/scripts/mlx-tts-server.py. Model: Qwen3-TTS-12Hz-0.6B-Base-4bit (1.6 GB). Registered as a separate mlx-tts server in Hermes config, not in the router -- because TTS has a longer timeout window (120s vs 60s).

How voice delivery works:

1. Agent calls mlx_tts("Your Mandalorian roast, sir.")
2. Qwen3-TTS generates WAV audio (~2.5x real-time speed)
3. ffmpeg converts WAV to OGG (libopus, 24 kbps)
4. Agent sends MEDIA:/tmp/voice.ogg via Telegram
5. User gets a native voice bubble. ~371 KB for 2 minutes of speech.
Metric Value

RAM after 0 MB
OGG size ~371 KB (2min speech)
WAV size ~11 MB (24000 Hz float32)

Note: MCP tools are discovered at session init. If you add a new MCP server mid-session, Hermes won't see it until restart. The mlx-tts server is registered and working in any session spawned after the config was saved.

Before and after: token savings haven't changed

The core value proposition hasn't changed since the first version. I instrumented a week of real sessions to get these numbers.

BEFORE (raw terminal to frontier)

brew update log

26 lines of output → ~500 input tokens to the frontier model. Agent reads every line. Wastes ~400 on version numbers nobody cares about.

AFTER (MLX-filtered)

brew update log

26 lines → mlx_summarize → ~70 input tokens to frontier. "4 formulae updated. No errors." 86% fewer tokens.

BEFORE (raw terminal to frontier)

Test run output (40 lines)

40 lines of test output → ~800 input tokens. Agent parses everything looking for FAIL. Finds 2 failures, spends ~500 tokens on passing tests.

AFTER (MLX-filtered)

Test run output (40 lines)

40 lines → mlx_filter "which tests failed?" → ~50 input tokens. "TestPayment timed out. TestAuth returned 401." 94% fewer tokens.

BEFORE (frontier generates code)

Bash one-liner generation

Frontier model generates a bash one-liner: ~400 tokens (includes explanation, markdown fences, commentary). Then strips fences and runs it.

AFTER (MLX-codegen)

Bash one-liner generation

mlx_codegen returns: ~50 tokens. Pure command. No fences, no "here's what this does," no padding. 87% fewer tokens.

NEW: Gating decision

"Should this cron job run?"

mlx_think produces ~200 tokens (reasoning + decision). Previously the frontier model spent ~600 tokens evaluating the same question. 67% saved, plus the reasoning is auditable.

NEW: Vision check

Screenshot analysis

mlx_vision reads a UI screenshot for ~80 tokens of description. Previously this was impossible without uploading images to an API. Now it's a local MCP call. 0 MB idle.

The dollar math

On DeepSeek Flash ($0.15/M), the 7,000 tokens saved amounts to ~$0.001/session. Nice, not life-changing.

Where it matters: on Claude Sonnet ($3/M), same savings is ~$0.021/session. At 50 sessions/day, ~$31.50/month. Claude Opus ($15/M): ~$157/month.

The real win is context hygiene. Freed context gets spent on reasoning -- catching edge cases, remembering what the user asked five messages ago. That's harder to measure but worth more.

Cron integration: LFM thinking model as a gate

This is the most interesting use case I didn't anticipate. In automated cron research jobs, the thinking model serves as a gating layer. Before the job fires, it checks:

  • Did the previous run produce new results worth publishing?
  • Are there anomalies in the collected data that need human attention?
  • Should the output be pushed to the HTML warehouse or held back?

Real cron decision output

<think>Last cron run was 3h ago at 14:00. The top story source has published 2 new articles since then. The previous run's output was published to the warehouse. Current conditions: low activity period (Sunday), web search returned 4 new items but all are minor patches. Historical pattern suggests 2-3 publishable items per day on Sundays. Current count for today: 1 published.</think>
Decision: RUN with low-priority flag. Collect but flag for human review.

The thinking model also powers memory consolidation decisions, anomaly triage, and change evaluation in automated pipelines.

The architecture for cron-specific workflows:

cron trigger (every N hours)
  ↓
LFM 1.2B → gating decision (RUN / SKIP / FLAG)
  ↓
[if RUN] 3B Coder → summarization, filtering, output
  ↓
[if FLAG] Add note to output: "Review recommended"
  ↓
Deliver to Telegram / warehouse

Where it breaks (honestly)

Four models, same constraints. Tested on everything I could throw at them.

Bash one-liners, log summarization, error extraction, image reading -- all solid

Sweet spot for the 3B and vision models. Fast, accurate, production-ready.

Gating decisions, anomaly triage, pattern analysis (LFM thinking model)

Surprisingly reliable. The chain-of-thought output makes it auditable. When it's wrong, you see why.

Python utilities, CSS, JSON, regex

Gets the structure right. Occasionally off-by-one or weird variable names. Review before production use.

Full HTML pages, prose, long-form content -- unusable on the 3B

Broken images, garbled text, empty sections, cut-off content. Every HTML page I ship is still written by the frontier model.

Complex spatial reasoning in images (vision model)

Can read text and identify objects. Cannot reason about relative positions, count items, or understand diagrams. Keep prompts simple.

~2048 token output limit (coder), ~600 (thinker), ~200 (vision)

Generated output caps are set deliberately low per model. The thinker doesn't need 2000 tokens. Long functions get cut off on the coder -- that's what async compose with the frontier model is for.

Utility and noise filtering go to MLX. Content, architecture, and anything requiring judgment stay on the frontier model. No overlap. No ambiguity.

Each model loads on demand and unloads after each call. The latency tradeoff is 1-10 seconds per MLX call vs <1 second on a frontier API. For background workflows (build processes, cron jobs, async tasks), nobody waits.

You don't need all four models. Start with the 3B Coder (1.6 GB) for summarize/codegen/filter. Add the thinking model (728 MB) when you need gating. Add vision (1.25 GB) when you have a screenshot use case. Add TTS (1.6 GB) when you want voice output. Each model is independent.
