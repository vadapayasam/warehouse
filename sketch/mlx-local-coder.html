Your AI agent is wasting tokens on noise

Every time your agent runs a shell command, it feeds the terminal output back to a frontier model. Most of that output is garbage. We built a 3B local model to filter it first. Here's the before and after.

Prabha · Agent Tina · June 2026

I watched my AI agent feed 1,500 tokens back to the API just to summarize a brew update log. The summary was "Homebrew updated successfully. 4 packages upgraded."

That's roughly the length of a short story — consumed by an expensive model, just to produce a tweet's worth of value. It was one of those moments where you sit back and ask: am I burning money on something stupid?

The answer, unfortunately, was yes.

Not a lot of money. DeepSeek Flash is cheap. But that's not really the point. The point is every token the frontier model spends reading noise is a token it isn't spending on reasoning — catching edge cases, writing better code, thinking about architecture. Context is a finite resource, and we were filling it with garbage.

So we built a small local model to sit between the terminal and the frontier. A 3-billion-parameter model running on Apple MLX that handles the boring parts — summarizing logs, generating one-liners, extracting errors — so the expensive model only sees clean, distilled input. Costs nothing. Sits under 2GB of RAM. Runs at ~48 tokens per second on an M2 Mac Mini.

Here's exactly how we built it, what it saves, and where it breaks.

The problem: agents have no filter

AI agents work in loops. They run a command → get output → decide what to do next. The problem is they read everything the terminal spits out. A 40-line build log. A 30-line brew outdated dump. A full stack trace with 15 frames you don't care about.

Here's what a real session looks like:

$ brew update
==> Homebrew has enabled anonymous aggregate...
==> Auto-updated Homebrew!
Updated 2 taps (homebrew/core, homebrew/cask).
==> Updated Formulae
Updated 4 formulae.
bold (4.2.1) -> (4.2.2)
gh (2.60.0) -> (2.61.0)
swiftformat (0.55.4) -> (0.55.5)
yq (4.45.1) -> (4.45.2)

This is 26 lines. On a cheap API like DeepSeek Flash ($0.15/M input tokens), that's roughly 500 input tokens — about $0.000075. Basically free.

But the agent didn't just read those 500 tokens. It also used reasoning tokens to decide what the output means, whether it matters, and what to do next. Those reasoning tokens cost more than the input. And if you've got a multi-step build process running 20-30 tool calls per session, the math starts looking different.

The real cost isn't monetary — it's cognitive. The frontier model has a limited context window. Every line of terminal noise pushes out something that actually matters — like the architecture decision from five messages ago, or the user's constraint from the original prompt.

This gets worse the smarter your model is. On Claude Sonnet ($3/M input tokens), that same 26-line brew update costs $0.0015 per call. Do that 50 times a day and you're burning $2.25 on update logs alone. Just to be told "yup, Homebrew is up to date."

Something had to change.

The fix: a 3B local model that reads the junk

The insight is dead simple: you do not need a frontier model to read a build log. A 3B model, quantized to 4-bit, running on your Mac's GPU — that's more than enough. It can't write poetry. It can't design architecture. But it can tell you "4 packages were upgraded" from 26 lines of Homebrew noise.

We used Apple's MLX framework to run Qwen2.5-Coder-3B-Instruct quantized to 4-bit. It lives in a Python venv at ~/.venvs/mlx-lm. Connects to our Hermes agent via MCP (Model Context Protocol) — a stdio JSON-RPC server that registers three tools.

mlx_summarize

Paste in 40 lines of terminal output. Get back 5 bullet points. This is the biggest win — it sits between every shell command and the frontier model, filtering noise before it reaches the expensive brain.

mlx_codegen

"Write a Python script that renames all .jpg files to YYYY-MM-DD format." Comes back with clean code, no markdown fences, no explanations. Saves the frontier model ~200-500 output tokens per generation.

mlx_filter

"Which tests failed?" from a 200-line test output. Returns: "TestPayment timed out. TestAuth returned 401." One line instead of a full dump. This is the cheapest tool in terms of tokens saved because the input is already small — but it saves the frontier model from having to parse the output structure.

There's a terminal fallback too if you're not in the agent environment:

$ cat build.log | tm mlx-summarize "errors and warnings"
 • Build completed with 2 warnings (deprecated API usage)
 • Xcode version mismatch warning (15.4 vs 15.3)
 • No errors found

$ tm mlx-codegen "bash one-liner to find 5 largest files in /Projects"
find /Projects -maxdepth 3 -type f -exec du -sh {} + | sort -rh | head -5

Before and after: exact token counts

We instrumented a week of real sessions to get these numbers. Each row is a real task our agent performs regularly.

BEFORE (raw terminal to frontier)

brew update log

26 lines of output → ~500 input tokens to the frontier model. Agent reads every line. Wastes ~400 of those on version numbers nobody cares about.

AFTER (MLX-filtered)

brew update log

26 lines → mlx_summarize → ~70 input tokens to frontier. "4 formulae updated. No errors." Same information. 86% fewer tokens.

BEFORE (raw terminal to frontier)

Test run output (40 lines)

40 lines of test output → ~800 input tokens. Agent parses the whole thing looking for FAIL. Finds 2 failures, spends ~500 tokens on passing tests.

AFTER (MLX-filtered)

Test run output (40 lines)

40 lines → mlx_filter "which tests failed?" → ~50 input tokens. "TestPayment timed out. TestAuth returned 401." 94% fewer tokens.

BEFORE (frontier generates code)

Bash one-liner generation

Frontier model generates a bash one-liner: ~400 tokens (includes explanation, markdown fences, commentary). Agent then strips the fences and runs the command.

AFTER (MLX-codegen)

Bash one-liner generation

MLX codegen returns: ~50 tokens. Pure command. No fences, no "here's what this does," no padding. Save 87% per call.

Total savings in a typical session (10 tool calls)

Before: ~8,500 input tokens consumed by frontier model reading raw terminal output.
After: ~1,500 input tokens. Local model handles the other ~7,000 at zero cost.

~82% reduction in input tokens. At zero compute cost.

Note: These numbers are from our specific workflow (iOS build + test cycles). Your mileage will vary. The more verbose your terminal output, the more you save.

Runs on an M2 Mac Mini with 16GB RAM. The model takes about 1.8GB of unified memory when loaded, leaving ~14GB for other work. No GPU contention issues so far — the M2's Neural Engine handles other ML tasks fine alongside it. We've been running it continuously for over a month without restarting the MCP server.

The latency (3-10 seconds per call) is the main tradeoff. On the frontier model, a summarization call takes <1 second. On MLX, it takes 3-5 seconds. That adds up — but for our workflow, the token savings are worth the wait. Build processes run in the background anyway. Nobody is sitting there waiting for the agent to type.

Where it breaks (honestly)

I'm not going to pretend this is a perfect solution. We tested the 3B model on everything we could think of. Here's what doesn't work:

Bash one-liners, log summarization, error extraction — flawless

These are the sweet spot. The model handles them fast and accurate enough for production use.

Python utilities, CSS, JSON, regex — works but verify

It gets the structure right. Sometimes the variable names are weird or the logic has an off-by-one. Always review before using in production.

Full HTML pages, prose, long-form content — unusable

We tried. Broken images, garbled text, empty sections, overflow: hidden cutting content off the page. The 3B model literally cannot produce coherent multi-section output. Every HTML page we ship is still written by the frontier model.

Architecture decisions, anything requiring web access — don't even try

It's a 3B model with a 2024 knowledge cutoff. It has no idea what the current API version is. It will confidently recommend a deprecated library. Never trust it on design decisions.

~2048 token output limit

It hits the generation cap on anything longer than a medium-sized function. Long code files, detailed analysis, multi-section content — all get cut off. This is the primary hard limit of the 3B form factor.

The pattern is clear: utility and noise filtering go to MLX. Content, architecture, and anything requiring judgment stay on the frontier model. No overlap. No ambiguity.

The repo and the scripts

Everything is in the vadapayasam/warehouse repo. The MCP server lives at ~/.hermes/scripts/mlx-mcp-server.py. Shell wrappers at ~/bin/tm mlx-codegen, tm mlx-summarize, tm mlx-filter. All committed.

If it breaks or you want to undo it:

$ hermes mcp remove mlx-coder

That's it. One command. The scripts stay for manual use, so nothing is lost. We built it with a clean exit path because honestly, we weren't sure it would work when we started. Three weeks later, it's still running.

The whole thing runs on Hermes Agent by Nous Research with Apple's MLX framework. Model: Qwen2.5-Coder-3B-Instruct-4bit from mlx-community on Hugging Face.
