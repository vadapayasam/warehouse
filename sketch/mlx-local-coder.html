MLX Local Coder

Zero-cost inference for your AI agent. Offload code generation, log summarization, and error extraction to a local model running on your Mac. Cuts input tokens by 60-80% on verbose tasks. Costs nothing in dollars.

June 13, 2026 · Prabha · Agent Tina

What this is

MLX Local Coder is a framework that runs a small 3-billion-parameter model locally on your Mac and connects it to your AI agent via MCP (Model Context Protocol). The big model handles planning, architecture, and content. The local model handles cheap stuff: short code, log summaries, and filtering noise.

The result: your frontier model spends context tokens on what matters, not on 200-line build logs or repetitive script generation. And the local model costs zero dollars per token.

The big model plans. The local model executes. This is the hybrid pattern.

We built this for a specific problem: our AI agent (Hermes) was burning input tokens reading verbose terminal output. A brew update result. A 40-line test run. A stack trace. Each of those costs 500-2000 input tokens when the frontier model reads them. Over a session, that adds up fast.

So we built a local MCP server that runs a quantized Qwen2.5-Coder-3B-Instruct-4bit model at ~48 tokens per second using Apple's MLX framework. It lives on an M2 Mac Mini. Uses about 1.8GB of RAM. And it costs nothing to run.

The biggest savings come from input token compression. When the agent runs a command that produces 40 lines of output, the frontier model normally consumes all 40 lines as input context. With MLX, the agent pipes those 40 lines through mlx_summarize and gets 6 clean bullet points. That's 500-2000 tokens saved per call.

Code generation saves output tokens instead. A bash one-liner from MLX costs ~50 output tokens. The same task on Claude Sonnet costs ~150-300 output tokens + input overhead. The dollar difference is small on cheap models but adds up fast on premium ones.

How it works

Three tools. One MCP server. One small local model. The architecture is dead simple:

You (Telegram) | v Frontier Agent (DeepSeek Flash / Claude / etc) Plans, decides, routes work | ├─────────────────────────────┐ | | v v mlx_codegen mlx_summarize mlx_filter "write a Python "summarize "extract errors script that..." this build log" from this output" | | v v Qwen2.5-Coder-3B-4bit ← ~48 tok/s on M2 Mac Mini 1.8GB RAM, zero API cost | v Returns clean result → Frontier continues with cleaner context

The MCP server runs via stdio transport. Hermes spawns it as a child process and communicates over JSON-RPC. Each tool is a separate handler that sends a prompt to the local model and returns the output. No network calls. No API keys. No rate limits.

mlx_codegen

Generate code from a prompt. Python scripts, bash commands, CSS snippets, JSON configs, regex patterns. Returns clean code with no markdown fences or explanations.

mlx_summarize

Condense verbose output into concise bullet points. Build logs, test results, file contents. The biggest token saver in practice.

mlx_filter

Extract a specific answer from noise. "Which tests failed?" "What error occurred?" "Did deployment succeed?" One line back, no parsing.

There's also a terminal fallback via tm scripts if the MCP tools aren't available:

tm mlx-codegen "bash one-liner to find 5 largest files"
cat build.log | tm mlx-summarize "errors and warnings"
cat test-out.txt | tm mlx-filter "which tests failed?"

What it handles. What it doesn't.

We tested the 3B model systematically. Some things it nails. Some things it catastrophically fails at. Knowing the split is the whole point of having this framework.

Task Verdict Details
Bash one-liners Excellent find /Projects -maxdepth 1 -type f -exec du -sh {} + | sort -rh | head -5 — correct first try, no commentary
Build log summarization Great 34-line brew update → 6 clean bullets with exact counts and error status preserved
Python utility scripts Good Short utility scripts work. Complex logic needs verification.
CSS/JSON/Regex Good Predictable outputs with clear structure. Same class as bash/Python.
Error extraction from logs Great "TestPayment failed because it timed out." — clean, specific, correct.
Full HTML pages with content Poor Broken image refs, garbled text, empty sections, layout bugs. Cannot handle long-form content synthesis.
Content writing / prose Fails 3B model cannot produce coherent multi-section narrative. Reads like garbled placeholder text.
Architecture decisions Fails No reasoning depth. Will confidently suggest wrong approaches.

The pattern is clear: utility code and log processing go to MLX. Content, architecture, and complex output stay on the frontier model.

How it runs

Model

mlx-community/Qwen2.5-Coder-3B-Instruct-4bit. Quantized to 4-bit via MLX. Runs in a Python venv at ~/.venvs/mlx-lm.

Hardware

M2 Mac Mini, 16GB RAM. Model takes ~1.8GB resident. Leaves ~14GB free for other work.

Speed

~48 tokens per second on M2. A 200-line build log summarization takes 3-5 seconds. Code gen takes 5-10 seconds.

Transport

MCP stdio server. Hermes spawns the Python script as a child process. JSON-RPC over stdin/stdout. No network, no auth, no latency beyond model inference.

The MCP server registers three tools that appear natively in the agent's toolbelt. The agent doesn't need to remember to route to MLX manually — when the tool is called, the MCP server handles the JSON-RPC dialogue, runs the model, and returns the result. The frontier model gets clean data back.

Problems. Real ones.

This framework is useful but it is not magic. It has sharp edges you should know about before trusting it.

HTML output is unusable

The 3B model cannot produce content-rich HTML. We tested it with a detailed spec including colors, layout, and content. The output had broken image references, duplicated words, empty sections, and a overflow: hidden that cut off content below the fold. The model's ~2048 token output limit and weak cross-reference reasoning make it incapable of synthesizing research into a multi-section page. Every HTML page we ship is still authored by the frontier model with the prabha-taste design gate. MLX does not touch HTML.

3B model can be confidently wrong

It will produce plausible-looking code that has subtle bugs. Variable name mismatches. Off-by-one errors. Incorrect API calls that look correct. You have to verify every output before using it in production. The model is fast but not reliable enough to trust unsupervised.

No web access

The local model cannot browse the web, fetch documentation, or check current API versions. It can only process what you put into its context window. If it needs to use a library version from 2026, it will hallucinate one based on its training cutoff. Always pair MLX output with verification against actual docs.

Latency adds up

Each call takes 3-10 seconds. This is fine for a summarization task during a multi-step build process. It is annoying for quick "what's this" lookups. The framework is not a drop-in replacement for the frontier model on latency-sensitive tasks. It is a deliberate routing choice that trades latency for token savings.

1.8GB RAM is permanent

The model stays resident once loaded. On a 16GB machine that's 11% of available RAM. This is fine on a desktop Mac Mini that runs 24/7. It would be annoying on a laptop you close and reopen. If you have other memory-heavy processes running, monitor the impact.

~2K token output limit

The 3B model's generation cap is roughly 2048 tokens. This is fine for bash one-liners, short Python scripts, and bullet-point summaries. It fails for anything longer. Long code files, multi-function scripts, or detailed analysis require the frontier model. This is the primary reason MLX cannot replace the frontier model for full-feature output.

The code

Every script is committed to the vadapayasam/warehouse repo. The MCP server is at ~/.hermes/scripts/mlx-mcp-server.py on the machine. The three Python scripts and shell wrappers all live under ~/.hermes/scripts/ and ~/bin/.

The framework is fully reversible. Disable it with a single command:

hermes mcp remove mlx-coder

Scripts stay for manual use. No data loss. No config side effects. We built it with a clean exit path because we tested it for a few days before committing to the pattern.

Build infrastructure when tokens are cheap. It pays for itself when you upgrade models.

That's the bet we're making. DeepSeek Flash is cheap right now. But the moment we switch to a better model, this framework starts saving real money. And since the local model costs nothing to run, the savings are pure margin.

When to route to MLX

After months of using this setup, the pattern is settled. Here's the decision tree the agent follows automatically:

Need something generated?

├── Full HTML page with research content?
│   → Frontier model + prabha-taste gate. MLX does not touch this.

├── Complex code or architecture decision?
│   → Frontier model only. MLX lacks reasoning depth.

├── Bash one-liner, alias, or Python utility?
│   → MLX. Returns clean code in 3-5 seconds.

├── CSS snippet, regex pattern, or JSON/YAML config?
│   → MLX. These are predictable structured outputs.

├── Verbose build log or test output to summarize?
│   → MLX. This is where the biggest token savings come from.
│      Saves 500-2000 input tokens per call.

├── Need to extract one specific error from noise?
│   → MLX. "TestPayment failed: timeout." One line back.

└── Content writing, long-form prose, or anything opinionated?
    → Frontier model. MLX cannot write coherently.

The rhyme is simple: frontier owns architecture and prose. MLX handles the utility pieces around the edges. No overlap, no ambiguity, no wasted tokens.
