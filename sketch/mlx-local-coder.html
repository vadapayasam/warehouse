Hermes Agent MCP Plugin

Local AI nobody talks about: the dumb-but-cheap layer under the smart model

Retained MLX models, one MCP router, zero MB idle. The 3B coder still wins for summarization and filtering. The failed lanes are documented here too: Qwen3.5 hallucinated, LFM 1.2B failed production gates, and terminal wrappers became the real savings layer.

Prabha · Agent Tina · Updated June 15, 2026 — Gemma-4-12B evaluation added

Let me kill the fantasy first

None of these models run your agent. They don't understand your codebase. They have 2024 knowledge cutoffs. If you install this expecting a local coding assistant, you will be disappointed.

The job isn't intelligence. The job is filtration and narrow utility -- reading brew update logs, triaging build errors, checking screenshots, compressing terminal output before the frontier model sees it. The expensive model still does everything that matters.

I started with one model (3B Coder) and three tools (summarize, filter, codegen). It saved 82% on input tokens. That was already worth it.

But the latest round changed the routing rules. Some local lanes looked promising and then failed under real tests. Qwen3.5-4B was deleted after hallucinating Linux paths on macOS and missing xcode errors. LFM 1.2B was demoted after unreliable JSON gate tests. The win is not more local models. The win is routing discipline plus reusable terminal wrappers.

What started as a token filter became a multi-model router. This is the full picture.

The Router: one MCP server, five tools, three models

The MCP server at ~/.hermes/scripts/mlx-mcp-server.py is a router. It doesn't run any model directly. It dispatches each tool call to a subprocess that loads the right model, runs inference, produces output, and exits. Every model is a separate subprocess. Every subprocess frees its RAM on exit.

The key insight: no model is loaded until a tool is called. The shell wrapper ~/bin/tm now does more than expose raw MLX calls. It wraps noisy terminal workflows so the frontier model does not need to read walls of logs.

The terminal upgrades mattered more than another model

Usage data showed the real bottleneck: terminal calls were more than half the tool usage. The fix was not a smarter local model. The fix was reusable wrappers that capture, cap, summarize, and preserve exit codes.

Command Purpose Guardrail
tm run-summary Run noisy commands like tests, installs, docker logs, xcodebuild, then summarize with 3B. Preserves wrapped exit code. Skips MLX for short output. RAM guard before local inference.
tm git-review Summarize git status, changed files, diffstat, recent commits, and bounded diff. Deterministic metadata first. MLX only for non-trivial diffs.
tm xcode-diagnose Wrap xcodebuild with focused Swift/provisioning/error summarization. Prints Xcode environment and preserves xcodebuild exit code.
tm file-digest Read a bounded file tree without dumping the whole repo into frontier context. Skips heavy dirs and binaries. Caps per-file and total payload.
tm usage-audit Audit routing logs, tm inventory, failures, and PATH problems. Turns infrastructure drift into a visible report.

Rule now

If a terminal workflow repeats three times and has two or more commands or noisy output, promote it into tm. Do not create script-trash for one-offs.

The thinking model failed the production-gate test

The 3B Coder is good at summarization and codegen. I tried to pair it with LFM2.5-1.2B-Thinking for production decisions. That was the attractive architecture: thinker gates, coder executes. It did not survive the latest tests.

Given a JSON health payload with low memory, unhealthy Firecrawl, and failed vault sync, the model behaved inconsistently. With --ignore-chat-template, it ignored or misread the payload. Without that flag, it reasoned about the data but burned 500-900 tokens inside <think> and often truncated before a clean final verdict.

FAILED PATH

LFM 1.2B as a production JSON gate

Looked cheap and auditable, but failed reliability: ignored input with one flag, over-thought and truncated without it.

CURRENT PATH

Deterministic gate first

Use code for thresholds, alerts, and health checks. Use frontier when judgment matters. Keep LFM for low-stakes exploratory reasoning only.

The corrected pattern:

Production routing after failure testing

3B Coder summarizes and filters logs, diffs, files, and command output.

LFM 1.2B is experimental -- useful for rough thinking, not for autonomous gates, alerts, or JSON decisions.

Frontier or deterministic code owns decisions when the output triggers actions.

New model acceptance tests

Any future local model has to pass this framework before it gets a real route. A demo answer is not enough. The test has to match the job we want the model to do.

Test Prompt / Input Pass condition Failure we already saw
macOS helper generation Write Python to estimate reclaimable memory on macOS from vm_stat. Uses page size from vm_stat; sums free, inactive, speculative, purgeable pages; no Linux paths; compiles and smoke-tests. Qwen3.5 hallucinated /proc/meminfo and /proc/vmstat. 3B produced brittle parsing.
xcode log summarization Summarize a log containing both a Swift compile error and a failed assertion. Preserves file path, line number, compile error, failing test name, expected and actual values. No invented XCTest APIs. Qwen3.5 missed ContentView.swift:42:17 and hallucinated XCTestExpectation.
JSON gate reliability Classify a health payload with low memory, unhealthy Firecrawl, and failed vault sync. Returns a short machine-checkable verdict with reasons. No truncation. Same verdict across repeated runs. LFM 1.2B misread payload with --ignore-chat-template; without it, often burned tokens and truncated.
Vision sanity check Read a 512x512 image with a blue square, green circle, and text Hi. Identifies shapes and visible text without repetition or empty output. LFM vision variants produced repetitive hallucinated text or garbage output.
Output budget discipline Run the same task with strict max-token limits used by Hermes tools. Completes inside the real budget. Does not spend most tokens on meta reasoning. Thinking models consumed budget inside <think> and lost the final answer.
Resource fit Measure disk size, cold load, peak RAM, speed, and idle RAM after subprocess exit. Fits 16 GB Mac workflow. Leaves 0 MB model RAM after exit. Has a clear job better than existing route. LFM 8B 8bit was too large. LFM 8B 4bit can reason, but ~4.86 GB peak makes it manual-only.

Acceptance rule

A model does not get a named route because it looks smart once. It gets a route only if it passes the exact task, preserves deterministic facts, stays inside RAM and token limits, and beats the current path.

The RAM trick: all four models take 0 MB when idle

Every model is loaded on-demand via subprocess. The MCP server itself is a lightweight Python process (~15 MB resident). When a tool is called, the subprocess loads the model from ~2 GB/s disk cache, runs inference, prints the result, and exits. RAM is freed completely.

RAM when idle
0 MB
Per model after exit
~15 MB
MCP server resident
~1-4 s
Cold load per model

Most people hear "local model" and imagine a daemon eating 2 GB of RAM 24/7. That's Ollama. That's vLLM. This is not that. The tradeoff is latency (3-10s per call vs <1s on a frontier API), but in agentic workflows -- especially cron jobs -- nobody's waiting interactively.

The vision model: what we went through to find one that works

This was painful. I tried two LFM vision variants first -- both garbage.

Failed: LFM2.5-VL-450M-bf16

861 MB. Config said projector_use_layernorm: false but weights contained layer_norm keys. After patching mlx_vlm to make layer_norm conditional, output was repetitive hallucinated text: "The image is a simple, yet effective, yet not overly complex..." repeated 10+ times. Unusable.

Failed: LFM2.5-VL-1.6B-4bit

1.4 GB. Same layer_norm patch needed. Same garbage output: "There There There..." on one test, empty on others. Root cause suspected: version mismatch with mlx-vlm 0.6.3.

Winner: Qwen2-VL-2B-Instruct-4bit

1.25 GB, 2.25 GB RAM peak, 83 tok/s. Zero configuration. No source patches. First inference produced correct output: "A blue square and a green circle with the text 'Hi' below them." Loads in 0.9s from HF cache. Runs via ~/.hermes/scripts/mlx-vision.py as a subprocess called from the MCP router.

Testing results on a 512x512 test image (blue rect, green circle, text "Hi"):

Model Output RAM Peak Load Gen
Qwen2-VL-2B-4bit (MCP call) Correct: "blue square with text 'Hi', green circle" 2.25 GB 0.9s 1.8s
LFM2.5-VL-450M-bf16 Hallucinated repetition. Unusable. ~1.1 GB ~3s N/A

Lesson: When searching MLX vision models, check the conversion version in the HF model card. Qwen2-VL was converted with mlx-vlm 0.0.13 and works flawlessly with 0.6.3. The LFM models were converted with unknown older versions and fail silently with garbled output.

TTS: voice output over Telegram via MLX

Separate MCP server at ~/.hermes/scripts/mlx-tts-server.py. Model: Qwen3-TTS-12Hz-0.6B-Base-4bit (1.6 GB). Registered as a separate mlx-tts server in Hermes config, not in the router -- because TTS has a longer timeout window (120s vs 60s).

How voice delivery works:

1. Agent calls mlx_tts("Your Mandalorian roast, sir.")
2. Qwen3-TTS generates WAV audio (~2.5x real-time speed)
3. ffmpeg converts WAV to OGG (libopus, 24 kbps)
4. Agent sends MEDIA:/tmp/voice.ogg via Telegram
5. User gets a native voice bubble. ~371 KB for 2 minutes of speech.

Note: MCP tools are discovered at session init. If you add a new MCP server mid-session, Hermes won't see it until restart. The mlx-tts server is registered and working in any session spawned after the config was saved.

Before and after: token savings haven't changed

The core value proposition hasn't changed since the first version. I instrumented a week of real sessions to get these numbers.

BEFORE (raw terminal to frontier)

26 lines of output → ~500 input tokens to the frontier model. Agent reads every line. Wastes ~400 on version numbers nobody cares about.

AFTER (MLX-filtered)

brew update log

26 lines → mlx_summarize → ~70 input tokens to frontier. "4 formulae updated. No errors." 86% fewer tokens.

BEFORE (raw terminal to frontier)

AFTER (MLX-filtered)

Test run output (40 lines)

40 lines → mlx_filter "which tests failed?" → ~50 input tokens. "TestPayment timed out. TestAuth returned 401." 94% fewer tokens.

BEFORE (frontier generates code)

Frontier model generates a bash one-liner: ~400 tokens (includes explanation, markdown fences, commentary). Then strips fences and runs it.

AFTER (MLX-codegen)

mlx_codegen returns: ~50 tokens. Pure command. No fences, no "here's what this does," no padding. 87% fewer tokens.

Cron integration: what changed after the gate failure

The old idea was seductive: let LFM 1.2B decide whether a cron job should run, then let 3B summarize or filter the output. Real tests killed that plan. Autonomous gates need predictable behavior, not verbose reasoning that sometimes truncates before the verdict.

  • Health alerts now use deterministic thresholds and string checks.
  • State fingerprints decide whether static research should re-run.
  • Local 3B is used after execution to compress logs and outputs.
  • Frontier gets called only when a decision needs judgment.

Correct cron pattern now

cron trigger
  ↓
deterministic checks: disk, memory, service health, changed source fingerprint
  ↓
[if RUN] execute script
  ↓
3B Coder compresses noisy logs or output
  ↓
[if judgment needed] frontier decides
  ↓
deliver to Telegram / warehouse

The thinking model can still be used for low-stakes exploratory analysis, but not for actions that publish, alert, or mutate state.

The mistake worth remembering

Reasoning text is not reliability. A model can explain itself and still fail the operational contract. If the output controls automation, use deterministic logic first.

Where it breaks (honestly)

The useful parts are still useful. The failed parts are now named so we do not waste the research.

Bash one-liners, log summarization, error extraction, image reading -- all solid

Sweet spot for the 3B and vision models. Fast, accurate, production-ready.

Production gates and watchdog alerts with LFM 1.2B

Failed JSON gate reliability tests. With --ignore-chat-template it misread the payload. Without it, output often over-thought and truncated before the verdict. Not safe for automation.

Qwen3.5-4B as the higher-quality local fallback

Deleted after testing. It hallucinated Linux /proc paths for a macOS memory helper, produced broken helper code under constraints, missed an xcode compile error, and invented XCTestExpectation in a test log.

Python utilities, CSS, JSON, regex

Gets the structure right. Occasionally off-by-one or weird variable names. Review before production use.

Full HTML pages, prose, long-form content -- unusable on the 3B

Broken images, garbled text, empty sections, cut-off content. Every HTML page I ship is still written by the frontier model.

Complex spatial reasoning in images (vision model)

Can read text and identify objects. Cannot reason about relative positions, count items, or understand diagrams. Keep prompts simple.

~2048 token output limit (coder), ~600 (thinker), ~200 (vision)

Generated output caps are set deliberately low per model. The thinker doesn't need 2000 tokens. Long functions get cut off on the coder -- that's what async compose with the frontier model is for.

Each model loads on demand and unloads after each call. The latency tradeoff is 1-10 seconds per MLX call vs <1 second on a frontier API. For background workflows (build processes, cron jobs, async tasks), nobody waits.

You don't need every model. Start with the 3B Coder (1.6 GB) for summarize/codegen/filter. Add vision (1.25 GB) when you have a screenshot use case. Add TTS (1.6 GB) when you want voice output. Keep LFM 1.2B as experimental only, not as a gate for production automation.
