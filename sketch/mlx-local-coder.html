Hermes Agent MCP Plugin

You don't want a local model. You want a token filter.

I put a 3B model in my AI agent to save tokens on noise. It can't write features, can't design architecture, and has a 2024 knowledge cutoff. But it dropped my frontier model's input by 82%. Here's exactly how and why.

Prabha · Agent Tina · June 2026

Let me kill the fantasy first

This 3B model will NOT run your agent. It will not understand your codebase. It will not make architectural decisions. It has a 2024 knowledge cutoff and will confidently recommend deprecated APIs. If you install this expecting a local coding assistant, you will be disappointed.

That's fine — because that's not the job. The job is reading brew update logs so your expensive model doesn't have to.

I watched my AI agent feed 1,500 tokens to a frontier model just to summarize a brew update log. The summary came back: "Homebrew updated successfully. 4 packages upgraded."

That's roughly the length of a short story — consumed by an expensive model — to produce a tweet's worth of value. It was one of those moments where you sit back and ask: am I doing something stupid?

The answer was yes. Not because of the cost — DeepSeek Flash is cheap. But because every token the frontier model spends reading noise is a token it isn't spending on reasoning. Context is a finite resource, and I was filling 30-70% of it with junk.

What I built: a Cascading architecture for Hermes Agent

I run Hermes Agent by Nous Research. It calls tools, runs commands, and feeds output back to the LLM. The problem is every tool call dumps raw terminal output into the frontier model's context. Over a 10-call session, that adds up to ~8,500 input tokens of noise.

I built an MCP (Model Context Protocol) server that registers three tools on my Hermes agent. All three run on a local 3B model via Apple's MLX framework:

mlx_summarize

Condenses 40 lines of terminal output into 5 bullet points. Sits between every shell command and the frontier model. This is where most of the savings come from.

mlx_codegen

Generates bash one-liners, Python scripts, CSS snippets — returns clean code with no markdown fences or commentary. Saves ~200-500 output tokens per generation vs the frontier model's verbosity.

mlx_filter

"Which tests failed?" from 200 lines of output → "TestPayment timed out. TestAuth returned 401." One line instead of a full dump.

I call this a Cascading architecture: the cheap local model handles narrow, well-defined tasks (summarization, filtering, codegen). The frontier model only sees clean, distilled input. The expensive brain spends its tokens on things that actually need intelligence.

There's a terminal fallback too if you're outside the agent:

$ cat build.log | tm mlx-summarize "errors and warnings"
 - Build completed with 2 warnings (deprecated API usage)
 - Xcode version mismatch warning (15.4 vs 15.3)
 - No errors found

$ tm mlx-codegen "bash one-liner to find 5 largest files in /Projects"
find /Projects -maxdepth 3 -type f -exec du -sh {} + | sort -rh | head -5

Most people hear "local model" and imagine a daemon running 24/7, eating 1.8GB of RAM they'll never get back. That's how Ollama works. That's how vLLM works.

This is not that. The MCP server loads the model from disk, runs the inference, and unloads it. Completely. When it's not being called, it takes exactly 0 MB of RAM. Zero. The next call reloads from disk in about 3-5 seconds (cold) or <1 second if the OS hasn't evicted the page cache yet.

On an M2 Mac Mini with 16GB RAM, this is the difference between "I need to keep 1.8GB pinned to a service I might not use this session" and "I use exactly 0MB until I decide to call it." You don't feel it. There's no background process to kill.

The problem: my agent had no filter

AI agents work in loops. Run a command → get output → decide what to do next. The problem is they read everything the terminal spits out. A 40-line build log. A 30-line brew outdated dump. A full stack trace with 15 frames you don't care about.

$ brew update
==> Homebrew has enabled anonymous aggregate...
==> Auto-updated Homebrew!
Updated 2 taps (homebrew/core, homebrew/cask).
==> Updated Formulae
Updated 4 formulae.
bold (4.2.1) -> (4.2.2)
gh (2.60.0) -> (2.61.0)
swiftformat (0.55.4) -> (0.55.5)
yq (4.45.1) -> (4.45.2)

This is 26 lines. On a cheap API like DeepSeek Flash ($0.15/M input tokens), that's ~500 input tokens — about $0.000075. Basically free.

But the agent didn't just read those 500 tokens. It also used reasoning tokens to decide what the output means, whether it matters, and what to do next. And if you've got a multi-step build process running 20-30 tool calls per session, the math changes.

The real cost isn't monetary — it's cognitive. The frontier model has a limited context window. Every line of terminal noise pushes out something that actually matters — like the architecture decision from five messages ago, or the user's constraint from the original prompt.

On Claude Sonnet ($3/M input tokens), that same 26-line brew update costs $0.0015 per call. Do that 50 times a day and you're burning $2.25 on update logs alone. On Claude Opus ($15/M), it's $11.25 per day. Just to be told "yup, Homebrew is up to date."

Something had to change.

The dollar math (honest)

If you're on a cheap provider like DeepSeek Flash ($0.15/M), the 7,000 tokens saved per session amounts to ~$0.001 in raw input cost. Nice, but not life-changing.

Where it matters: on Claude Sonnet ($3/M), that same savings is ~$0.021/session. At 50 sessions/day, that's ~$31.50/month recovered. On Claude Opus ($15/M), it's ~$157/month.

But the real win is context hygiene. That 7,000 tokens of freed context space gets spent on reasoning — catching edge cases, thinking about edge-to-edge layouts, remembering what the user asked five messages ago. That's where the value lives, and it's harder to measure.

Note: These numbers are from my specific workflow (iOS build + test cycles on Hermes Agent). Your mileage will vary. The more verbose your terminal output, the more you save.

Where it breaks (honestly)

I'm not going to pretend this is a universal solution. I tested this 3B model on everything I could throw at it. Here's what doesn't work:

Bash one-liners, log summarization, error extraction — flawless

These are the sweet spot. Fast, accurate, production-ready. Use without hesitation.

Python utilities, CSS, JSON, regex — works but verify

Gets the structure right. Occasionally off-by-one or weird variable names. Always review before using in production.

Full HTML pages, prose, long-form content — unusable

Broken images, garbled text, empty sections, cut-off content. Every HTML page I ship is still written by the frontier model.

Architecture decisions, anything requiring web access — don't even try

It's a 3B model with a 2024 knowledge cutoff. It will confidently recommend a deprecated API. Never trust it on design decisions.

~2048 token output limit

Hits the generation cap on anything longer than a medium function. Long files, detailed analysis, multi-section content all get cut off. This is the primary hard limit of the 3B form factor.

The pattern is clear: utility and noise filtering go to MLX. Content, architecture, and anything requiring judgment stay on the frontier model. No overlap. No ambiguity.

The model loads, runs the inference, and unloads. You don't lose 1.8GB to a daemon that might never be called this session. On an 8GB Mac, this means you can use the local model when you need it and have every byte back the moment it's done.

The latency tradeoff is real: 3-10 seconds per MLX call vs <1 second on a frontier API. But for my workflow, build processes run in the background anyway. Nobody's sitting there waiting.

Setup

The MCP server lives at ~/.hermes/scripts/mlx-mcp-server.py. Shell wrappers at ~/bin/tm mlx-codegen, tm mlx-summarize, tm mlx-filter. All in the vadapayasam/warehouse repo.

To remove it:

$ hermes mcp remove mlx-coder

One command. The scripts stay for manual use.

Built on Hermes Agent by Nous Research with Apple's MLX framework. Model: Qwen2.5-Coder-3B-Instruct-4bit from mlx-community on Hugging Face.
