Beyond the Dashboard

The agent stack (Hermes + OpenClaw) has the legs. What it doesn't have is a control surface worthy of the agents living on it. Web UIs are dead. This is what replaces them.

Research · June 2026 Stack: Hermes Agent · OpenClaw · cmux · MCP/A2A

One-Liner Verdict

The missing layer in every agent stack isn't another framework - it's a unified ambient control surface that lives where the user already is (terminal, chat, menu bar, voice) and abstracts away whether the agent underneath is Hermes, OpenClaw, or custom. Telegram is uncertain. Discord isn't permanent. The control layer needs to be platform-agnostic, ambient, and intervention-native - a conductor, not a dashboard.

Landscape: The Agents We Have

The debate between OpenClaw (~35% community) and Hermes (~30%) dominates the agent space in 2026. ~20% run both. Neither has a communication layer that works across the other.

Protocols That Already Exist

MCP (Anthropic)

Vertical: Agent → Tool

Standardized data/tool interface. Every agent speaks it. But it's agent-to-tool, not agent-to-agent.

A2A (Google)

Horizontal: Agent ↔ Agent

Agent-to-agent communication protocol. Handles task delegation, status reporting, capability discovery.

ACP (OpenAI)

Coordination: Within-org agents

Agent Coordination Protocol. Focused on enterprise agent fleets with security/audit layers.

A2UI (OpenClaw)

Agent → Generated UI

Declarative framework: agents emit HTML with A2UI attributes. Canvas server pushes via WebSocket. Rich but chat-embedded.

The cmux Pattern (Closest Precedent)

cmux is a native macOS terminal built on Ghostty/Swift/AppKit that solves the "black box agent" problem with a vertical sidebar showing agent status, progress bars, workspace states, and native macOS notifications. Agents control the terminal to report to the human via `notify`, `set-status`, `set-progress`. It's agent-agnostic - works with Claude Code, Codex, OpenCode, Gemini CLI. This is the closest thing to the paradigm shift, but it's coder-only - no OpenClaw/Hermes integration, no cross-agent orchestration.

The Missing Layer

Here's what doesn't exist yet - and this is the product opportunity:

No Unified Agent Registry

If you run both Hermes and OpenClaw (20% of users do), there's no single place that knows all your agents. Each framework has its own idea of what agents exist.

No Ambient Awareness

You have to actively check agents. Open a chat. Open a web UI. Open a terminal. No peripheral-vision interface that lets you know "something needs attention" without demanding focus.

No Cross-Agent Intervention

If Hermes agent Tina hits a bug and OpenClaw agent Claw can help, there's no way for the systems to coordinate. The user has to manually context-switch between runtimes.

No Multi-Modal Control Surface

Telegram is uncertain. Discord isn't permanent. Web UIs are dead. Voice isn't there yet. The ideal control surface adapts to where the user is - terminal when coding, voice when driving, chat when on phone, menubar glance when at desk.

No Conductor / Meta-Agent Pattern

Nobody has built the agent that manages agents. An orchestrator that abstracts away which runtime is underneath and presents a unified interface to the human.

Product Concepts

These are buildable. Grounded in what exists. Ordered by practicality for Prabha's stack.

🚀 Build First

A3P - Ambient Agent Awareness Protocol

A lightweight message bus that both Hermes and OpenClaw agents publish to. The foundation everything else sits on.
Why this exists

Before you can build a control surface, agents need a common language to report status. A3P is a minimal JSON schema over Unix socket or local HTTP. Hermes publishes `task_start`, `task_complete`, `error`, `needs_review`. OpenClaw same schema. Any surface subscribes.

Schema (one message type):
{ type, agent_id, agent_runtime, session_key, status, summary, payload?, ts }

Implementation path: Unix domain socket listener in a background Hermes process + OpenClaw gateway plugin. ~200 lines each. No dependencies.

Unix socket JSON Plugin 200 LOC
🚀 Build Second

TINA - Terminal Integrated Agent Navigator

A macOS terminal-sidecar app that is NOT a web UI. It's a persistent tmux session with agent concierge - the place you glance at to know what's running.
Why it's different from web UIs

No browser tab, no Electron, no React. Just a self-healing tmux session with dedicated panes:

• Left sidebar: Agent registry (Hermes + OpenClaw + Codex) - color-coded status dots
• Main pane: Real-time activity log - every agent's output streams here
• Bottom bar: Quick command line - `/kill debug-agent`, `/focus hermes`, `/pause all`
• Floating alert: macOS notification banner on agent errors

Implementation: Bash/tmux wrapper + A3P subscriber via `socat`. Opens in your terminal instantly. Survives reboots. CLI-native, scriptable, SSH-able. 0 latency - no HTTP, no serialization overhead.

tmux Bash Unix socket macOS SSH-ready
🔧 Build Third

Conductor Agent - One Agent to Rule Them All

A meta-agent running inside Hermes that knows about every agent in your stack. You talk to ONE agent. It delegates to the right runtime.
Why this matters

The 20% of users running both Hermes and OpenClaw face a cognitive tax: "which agent should I talk to for this?" The Conductor removes that. It maintains a registry of agent capabilities, routes tasks, aggregates status, and intervenes when something goes wrong.

Capabilities: Agent registry (discovers every agent via A3P), task routing (delegates to the right runtime), unified status ("/status" returns everything running across all runtimes), intervention (/kill task-xyz kills it regardless of runtime), daily standup summaries.

Implementation: Hermes skill + OpenClaw gateway plugin. Uses existing session tools (sessions_send, sessions_spawn, sessions_kill) on the OpenClaw side and delegate_task on the Hermes side. Wraps both in a single chat interface.

Hermes skill OpenClaw plugin A3P Meta-agent
📱 Build Fourth

Ambient Agent Bar - macOS Menubar Pulse

A SwiftUI menu bar app showing agent ecosystem state in a single glance. Colored dot. No interface chrome. Ambient.
Why not a dashboard

You don't open a dashboard. The dashboard comes to you. A 1px colored dot in the menu bar: green (idle), blue (working), yellow (needs review), red (failed). Click it - dropdown shows running tasks, recent completions, error count. Click "kill all" to emergency-stop everything.

Implementation: SwiftUI AppKit wrapper. Subscribes to A3P via local socket. Uses Swift Concurrency (async/await) for zero-overhead polling. ~400 lines of Swift. Install via `brew install agent-bar`.

Prabha advantage: You're an iOS dev. This is SwiftUI on macOS. You can ship this in a weekend.

SwiftUI AppKit Menu bar macOS A3P
💡 Long-term Vision

Conductor Voice - "Hey Tina" Across the Stack

Voice-wake interface for the entire agent estate. No typing. No opening apps. Just speak.
Why this is the endgame

"Hey Tina, what's running?" → "Two agents active: the MSP research cron finished at 2:14pm, and Codex is still working on the auth refactor, about 60% done."
"Hey Tina, stop everything and summarize what I missed." → kills non-essential processes, collects output, speaks back a 30-second briefing.

Implementation: macOS speech recognition (SFSpeechRecognizer) → Conductor Agent → local TTS output. Uses the same Conductor backend - voice is just another input surface. The control plane is the same; the interface changes.

Build Order

Week 1: A3P Protocol

Unix socket listener + publisher. Hermes publish hook (background cron writes to socket). OpenClaw gateway plugin. 200 lines each runtime. Everything downstream depends on this.

Week 2: tmux Control Plane (TINA)

Persistent tmux session with A3P subscriber. Sidebar pane showing agent status. Bottom command bar for intervention. macOS notification integration via terminal-notifier.

Week 3: Conductor Hermes Skill

Hermes skill that discovers agents via A3P, maintains registry, routes tasks, aggregates status, handles interventions. Reuses existing Hermes skills infrastructure.

Week 4-5: macOS Menu Bar App

SwiftUI menu bar app with A3P subscriber. Colored dot status indicator, dropdown with running tasks, quick actions. Prabha's iOS experience makes this fast.

Long-term: Voice Interface

macOS speech recognition → Conductor Agent → TTS output. "Hey Tina" becomes the universal interface for the entire agent stack.

Why This Works for You

You already run the stack. Hermes (Tina) on Mac Mini. OpenClaw interest via MSPClaw. iOS development background. Telegram and Discord as current surfaces but you want something better.

You don't need another framework. You need the glue between what you already have. A3P + TINA + Conductor is ~1000 lines of code total, leveraging existing infrastructure.

The timing is right. OpenClaw and Hermes can exchange signed messages (2026 interop). A3P is the natural next step - a unified event bus that both runtimes speak. Nobody has shipped this yet. The community split (~20% run both) means there's a hungry audience.

This isn't a web UI. Terminal-native, tmux-native, menu-bar-native, voice-native. No Electron. No React. No login page. The control surface lives where you already are.
