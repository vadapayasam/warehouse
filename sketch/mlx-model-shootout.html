3B Never Needed Backup

We pitted Qwen2.5-Coder-3B against Qwen3.5-4B and Qwen3.5-9B in a three-round MLX benchmark on a Mac Mini M2. The smallest model won every round.

Why We Did This

We run local LLMs via MLX on a Mac Mini M2 with 16GB RAM. The setup is deliberately lean: load a model, generate output, unload to 0MB. No persistent server, no GPU allocation fighting with other apps, no idle VRAM tax. Every model must earn its place in the rotation.

For months, Qwen2.5-Coder-3B-Instruct-4bit has been our daily driver. It handles code generation, HTML prototyping, and text filtering without drama. Then Qwen3.5 dropped with 4B and 9B variants. Newer must be better, right?

We tested that assumption.

The Setup

Hardware: Mac Mini M2, 16GB unified memory
Software: MLX (mlx_lm generate), on-demand subprocess pattern
Models: Qwen2.5-Coder-3B-Instruct-4bit, Qwen3.5-4B-OptiQ-4bit, Qwen3.5-9B-OptiQ-4bit
Constraints: --max-kv-size 2000, --max-tokens 1024, --temp 0.0
Method: Each model runs three tasks. RAM measured before and after. No model stays loaded between runs.

Round 1: Python Codegen

Prompt: "Write a Python function that takes a list of file paths and returns a dictionary mapping file extensions to their counts. Handle hidden files and files without extensions."

WINNER 3B - 9.2s, 1.9GB RAM

Clean function. Correct edge case handling. Example usage included. Delivered in under 10 seconds.

Complete, functional, and fast. The layout was basic (flex-wrap cards) but it shipped.

Round 3: Text Summarization

Prompt: "Summarize this build failure report in bullet points. Focus on errors and warnings."

WINNER 3B - 4.2s, 1.9GB RAM

Five clean bullet points. Done in 4 seconds. No ceremony.

This round exposed the fatal flaw of Qwen3.5 for utility tasks. The 4B produced 2,061 characters for a task that needed 286. It printed its entire internal monologue: "1. Analyze the request... 2. Analyze the input... 3. Draft the summary... 4. Refine..." before finally outputting five bullet points. The 9B was even worse - consumed all 1,024 tokens on thinking and produced zero usable output.

The Qwen3.5 Problem: Thinking Mode

Both Qwen3.5 variants suffer from the same architectural quirk: they embed chain-of-thought reasoning inside their output. Every query triggers <think> blocks that consume 40-80% of the token budget before producing the actual response.

This is useful for complex math or logic puzzles. It is wasted on code generation (where the code itself is the output), HTML prototyping (where we need the finished markup, not the design decisions), and summarization (where we need the summary, not the analysis of how you wrote it).

The 4B spent 590 tokens thinking about how to write 5 bullet points. That's 5x the content of the actual output. On every single call.

Verdict

The 3B wins on every axis that matters for our use case:

  • 3-10x faster than the 4B, 4-20x faster than the 9B
  • 1.9GB vs 3.5GB vs 7.4GB - the 3B leaves room for other apps
  • No thinking noise - every token is output, not internal monologue
  • Reliable - never hit the output cap on our tasks
  • Complete - no truncated HTML, no half-written code

The 4B deserves an honorable mention for HTML quality - its CSS was genuinely better. But a beautiful truncated page is still a broken page. The 9B was a non-starter: 7.4GB RAM, truncated at every turn, and completely unusable for summarization.

What We Kept

Qwen2.5-Coder-3B-Instruct-4bit stays as the default MLX model for local inference. The 4B was downloaded, tested, and removed. The 9B never made it past the benchmark - 7.4GB of RAM on a 16GB machine is not negotiable.

The MLX on-demand subprocess pattern works: every model unloads to 0MB after each call. No VRAM fragmentation, no daemon management, no conflicts with Docker or Xcode. On a Mac Mini with finite memory, this is the right architecture.

Methodology Notes

All benchmarks were run on the same Mac Mini M2 (16GB) at idle (no Docker, no Xcode, no heavy browser). Each model was loaded fresh per task via mlx_lm generate subprocess. RAM was measured via psutil before and after each call. Generation speed excludes prompt processing time. Full benchmark code and raw output are available in the warehouse repo.

Models tested: Qwen2.5-Coder-3B-Instruct-4bit (mlx-community), Qwen3.5-4B-OptiQ-4bit (mlx-community), Qwen3.5-9B-OptiQ-4bit (mlx-community). All models quantized to 4-bit. Default MCP model remains 3B.

Built on a Mac Mini M2 in Chennai. No analytics, no tracking, no JavaScript. This is the Way.
