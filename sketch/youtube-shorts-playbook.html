1. YouTube Shorts Algorithm Mechanics

How It Actually Works (Not Marketing Speak)

YouTube Shorts uses a hybrid recommendation system different from long-form:

  • Session-based recommendations: Shorts are served in infinite scroll sessions, not search. The algorithm optimizes for session retention — keeping users in the Shorts feed.
  • SWF (Shorts Watch Feed): Primary traffic source. Videos are shown to small test pools first (50-500 views), then scaled if engagement passes thresholds.
  • Cross-over potential: High-performing Shorts can get pushed to long-form subscribers and vice versa.

Key Ranking Signals (In Priority Order)

Signal What Counts Target
AVD % Average View Duration % of video watched, not seconds 80%+ (higher = better)
Loop Rate Users rewatching immediately 15%+ loop rate
Engagement Velocity Likes/comments in first 24hrs 5%+ engagement rate
Share Rate Saves/shares per impression 1%+ share rate
CTR Thumbnail/title clicks 4-10% from browse
💡 Dev-Specific Insight: Dev content often has higher share rates than entertainment because developers save tutorials for later. YouTube treats saves as high-intent signals.

The "Goldilocks Zone" for Shorts Length

Research from top dev creators + YouTube's own data:

  • 15-22 seconds: Highest completion rates, best for simple tips/hacks
  • 25-34 seconds: Sweet spot for mini-tutorials, problem→solution narratives
  • 45-58 seconds: Max for complex explanations, build breakdowns
  • Avoid: 30-44 second range (algorithm dead zone — long enough to lose attention, short enough to feel rushed)

3. Hook Structures That Actually Work

First 1-3 seconds determine 80% of your reach. Here are proven formulas for dev content:

Formula 1: The "Impossible" Hook

"I built [X] in [unbelievably short time]"
"This [feature] took 20 minutes. Here's how:"

Why: Pattern interrupt. Developers are skeptical by nature — claiming speed/efficiency triggers "prove it" response.

Formula 2: The Controversy Hook

"Stop using [popular tool]. Do this instead."
"Your [X] is wrong. Here's what Apple actually recommends:"

Why: Forces viewer to prove you wrong. High comment engagement.

Formula 3: The "Steal This" Hook

"Steal this [component/code pattern] for your next app"
"Free SwiftUI component — just copy this:"

Why: Implicit value proposition. Saving = engagement.

Formula 4: The Reality Hook (Build-in-Public Specific)

"Day [X] building [app]: [unexpected thing happened]"
"Just lost [user metric]. Here's the fix:"

Why: Authenticity + narrative = relationship building. Not transactional like tutorial content.

Formula 5: The Visual Hook

[Show janky/broken UI first 0.5s] → "Fixed this animation"
[Show before/after split screen instantly]

Why: Visual pattern interrupt. Works even without sound.

Hook Type Avg Retention Best For
Impossible/Timeline 85-92% Tutorials, builds
Controversy 70-80% Opinions, commentary
Steal This 90-95% Code snippets, components
Reality/Process 75-85% Build logs, updates

5. Timing & Pacing Specifications

Optimal Length by Content Type

Content Type Target Length Max Length
Quick tip/One-liner 12-18s 20s
Code walkthrough 22-28s 35s
Build breakdown 30-40s 50s
Day wrap-up 25-35s 45s

Pacing Rules

  • Cut every 2-3 seconds: If a shot lasts longer, something should change (zoom, highlight, new element)
  • Text appears at 0.5x speed: Give viewers time to read — but never full duration
  • End at 95% watched: Last 5% is your CTA/follow prompt
  • Audio pacing: 140-160 words per minute maximum for clarity

The 3-Second Rule Breakdown

0.0-0.5s: Visual hook (motion, contrast, text pop)
0.5-1.5s: Context established (what is this about?)
1.5-3.0s: Promise made (why keep watching?)

Text Overlay Best Practices

  • Font: System fonts (SF Pro, Inter) for dev content — matches aesthetic
  • Size: Minimum 40pt for mobile readability
  • Position: Safe zone is middle 60% of vertical frame (avoid bottom 15% for UI elements)
  • Animation: Pop in (scale 0→1) or type-on effect — never static for full duration
  • Density: Max 6 words per text element. One concept per overlay.

B-Roll Patterns That Work

For dev content specifically:

  • Typing shots (hands on keyboard) — humanizes the code
  • Terminal/commands running — shows "real work"
  • Simulator with actual finger interactions — makes it tangible
  • Git commits timeline — visualizes progress
  • Face reactions (small corner overlay) — builds parasocial connection

7. AI Content Policy & Disclosure

YouTube's Official Stance

As of 2024, YouTube requires:

  • Disclosure of AI-generated content that shows realistic altered or synthetic imagery
  • No special disclosure needed for: AI-assisted editing, AI-generated code, AI text-to-speech (if clearly synthetic)
  • Required disclosure for: AI-generated faces/voices that could be mistaken for real, AI-generated screen recordings that mimic real apps

Algorithm Penalties

YouTube has NOT confirmed algorithm penalties for AI content. However:

  • Engagement quality matters more than origin: AI content that gets high retention performs identically to organic content
  • Misuse risk: Accounts spamming low-quality AI content face throttling
  • Recommendations from creators: Mix AI-assisted with genuine content (70/30 split recommended)

Build-in-Public AI Safeguards

For your content specifically:

  • Your Hermes automation is assistance, not generation — no disclosure needed
  • AI voiceovers: Use ones that sound synthetic (ElevenLabs "Josh" etc) or disclose
  • AI code suggestions: Fine to show, but clarify if code wasn't hand-written
  • AI thumbnails: No disclosure needed

Content Decision Tree

Before posting, ask:

  1. Would I save this to reference later? (Value check)
  2. Does this show process or just result? (Authenticity check)
  3. Would another dev DM me about this? (Shareability check)
  4. Is there an emotional hook? (Engagement check)
  5. Can someone learn something in <30s? (Shorts format check)
