MLX routing audit · June 15, 2026

Is the 0 MB local model router paying off?

Short answer: architecturally yes. Financially, not yet. The setup is sound, but usage is too low and logging is incomplete. Keep the MCP path. Do not pin models in RAM by default.

Audited on Mac Mini M2, 16GB RAM. Current chat runtime: GPT-5.5 via OpenAI Codex. Hermes config still lists DeepSeek Flash as the default model, which means this session is running under an override.

The verdict

Keep the 0 MB MCP design.

It is the right default for a 16GB machine. The models are burst tools, not residents. Pinning Qwen2.5-Coder-3B all day would burn roughly 1.9GB for occasional utility work.

Keep MCP Fix logging Do not always-load

Payoff status

Current measured savings are tiny: 2,621 input tokens and 2,055 output tokens routed locally across 10 logged calls. On DeepSeek Flash pricing, that is only $0.001626.

But this is the wrong scoreboard alone. The real benefit is context hygiene, fast local utility work, and a routing habit that becomes valuable when the frontier model is expensive, like GPT-5.5, Claude Sonnet, or future premium coding models.

What exists now

The setup is already more than a toy. It has multiple routing surfaces, a model zoo, tracker logs, and terminal wrappers. The weak part is not capability. The weak part is discipline and instrumentation.

What is not paying off yet

Usage is too low

Only 10 logged calls. The local router cannot save meaningful money if it is not used aggressively on build logs, file scans, session compression, cron gates, and terminal noise.

MCP logging is manual

The terminal wrappers log automatically. The MCP server code currently does not call the route tracker inside handle_tool_call. That means agent-side usage can be undercounted unless logged manually.

Duplicate MCP servers exist

Two current MCP server processes and two old Sunday MCP server processes were visible. They are small, but stale duplicate server processes are sloppy and should be cleaned.

Local reasoning can hallucinate

A test mlx_think audit suggested fake alternatives like TinyBench and DistilBench. That is proof: use LFM for narrow gates, not strategic architecture verdicts.

0 MB MCP vs always-loaded small model

Default answer: keep 0 MB MCP. Always-loaded only makes sense during a known local-heavy session, not as the system baseline.

Batch-mode compromise

Add a manual tm mlx-warm-session 30m later if needed. It can start a persistent local server for a short window, route many summarize/filter/codegen calls through it, then auto-kill. That gives speed when you are doing heavy local work without permanently taxing RAM.

Optimization plan

1

Patch MCP auto-logging

Add tracker logging inside mlx-mcp-server.py and mlx-tts-server.py. Manual logging is a footgun. If it is not automatic, the dashboard lies.

2

Add a hard routing checklist to Tina workflow

Before frontier reads noisy output: summarize locally. Before frontier parses a specific value: filter locally. Before cron runs: think locally. Before image reasoning: vision locally.

3

Promote summarize as the main saver

Summarization is where local models actually pay. Every build, test, git diff, install log, and long file excerpt should be compressed before frontier reasoning.

4

Restrict filter to exact extraction

mlx_filter is not semantic search. If the answer is not literally present in the input, skip it and use summarize or frontier. Stop burning failed calls there.

5

Wire Qwen3.5-4B as an optional specialist

Not always-on. Not default. Add an explicit tool later for structured JSON or higher-quality local summaries when 3B is too weak and frontier is overkill.

6

Clean stale processes and duplicate 8B caches later

Do not start with disk gardening. First fix routing behavior. Then reclaim the redundant LFM 8B variants if they are not part of an active benchmark.

Operating rule from here

Use frontier for

  • Planning and architecture
  • Public writing and HTML artifacts
  • Complex code review
  • Judgment requiring current web context
  • Anything where hallucination is expensive

Use local MLX for

  • Terminal output compression
  • Exact error extraction
  • Small utility code snippets
  • RUN or SKIP gating
  • Screenshot text and simple UI reading

Final call

The setup is not a money printer yet. It is a muscle. Right now the muscle is underused. Keep the MCP architecture, fix auto-logging, route more noisy work locally, and only use always-loaded models as a temporary batch mode.
