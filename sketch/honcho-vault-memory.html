V Layer 1: The Vault

Obsidian-based knowledge management at ~/Documents/hermes-vault/ (symlinked to ~/.hermes/obsidian-vault/). File-based, curated, permanent. This is where structured knowledge lives: MOCs, wiki notes, finance portfolios, design tokens, research artifacts.

How It Works

  • Raw inbox (raw/) - Daily drops, transcripts, screenshots, voice memos land here first
  • Processing pipeline - Agent distills raw into atomic wiki notes with frontmatter, tags, and wikilinks
  • Maps of Content (wiki/MOC-*.md) - Domain entry points: Agent-Tina, Finance, Design, Research, iOS
  • Daily digest - Cron at 1800 IST processes overnight files, generates digests, appends to monthly synthesis
  • REST API - Obsidian Local REST API on 127.0.0.1:27124, Bearer auth, enables agent reads/writes without opening the app

H Layer 2: Honcho

Self-hosted memory/RAG stack at localhost:8000. PostgreSQL + pgvector backend, OpenAI embeddings (text-embedding-3-small). Conversation-native - learns from what we say, not from what we write. Auto-injects relevant context into every agent turn.

How It Works

  • Auto-injection - Every session gets context from Honcho (session summaries, peer cards, surface patterns)
  • Dialectic - Background reasoning layer that connects sessions. Disabled (cadence=0) to save ~$9/mo in LLM costs
  • Deriver - Background processor that builds embeddings and semantic indexes from conversation history
  • Peer cards - Curated fact lists about the user (name, role, preferences, patterns) maintained automatically

Natural language questions over stored context. Depth levels from minimal to max. Costs more (Honcho LLM call).

honcho_conclude

Persist durable facts across sessions. "User prefers screenshot-centric QA." Never save temporary task state.

Synthesis Workflow

The combined flow when Agent Tina answers a cross-domain question:

1. Trigger: User asks a cross-domain question
   ("Should I raise WhatPay pricing?")

2. Honcho reasoning (medium/high):
   - Pull peer card for monetization preferences
   - Search past mentions of "pricing" + "WhatPay"
   - Synthesize patterns from conversation history

3. Vault lookup:
   - Check wiki/MOC-iOS for app status
   - Check wiki/finance/ for portfolio context

4. Deliver: Blended answer with sources cited
   ("Based on RevenueCat research [vault] + your past
    preference for testing before shipping [honcho]...")

5. Conclude (if new insight):
   - Save durable fact to Honcho for next session

Agent Tina Integration

Both memory systems are wired into Agent Tina's operating loop. Every task starts with a vault injection check and includes Honcho context. The flow is automatic:

1

Load vault-injection Skill

Triggers on task start. Extracts domain terms (finance, iOS, design), reads the relevant MOC, queries vault REST API for matching notes.

2

Receive Honcho Context

Every turn has auto-injected context: peer card, session summary, recent patterns. No explicit query needed.

3

Choose Memory Layer

Based on the question, pick the right tool. Honcho for preferences/patterns. Vault for structured data. Both for synthesis questions.

4

Save New Knowledge

Durable facts go to Honcho (honcho_conclude). Wiki notes go to the vault (write_file). MOCs updated when domains change.

Memory vs Profile Distinction

Agent Tina maintains two separate memory stores alongside the vault and Honcho:

  • Memory (memory tool, target='memory') - Agent's personal notes. Environment facts, tool quirks, conventions. Things the agent needs to remember about how to operate.
  • User Profile (memory tool, target='user') - Who the user is. Role, preferences, style, corrections. Things that stay stable across sessions.

These are compact, injected-every-turn stores. They do NOT replace the vault or Honcho - they're the agent's operating memory, not the knowledge base.

Infrastructure & Costs

Both systems run locally on a Mac Mini. No cloud dependencies (except Honcho's OpenAI embeddings at ~$0.02/M tokens).

Vault (Obsidian)

Free

Honcho API

~$1/mo

Honcho DB

Free (local)

PostgreSQL with pgvector extension. Stores session data + vector embeddings for semantic search.

Honcho Deriver

Included

Background processor that builds embeddings and generates session summaries. Low CPU, runs on idle.

Cost History

Honcho was burning ~$10/mo on Claude Sonnet 4 for dialectic analysis every 3 messages. Switched to Claude Haiku 4.5 and set dialectic_cadence: 0. Savings: ~90%. The dialectic processor is now disabled entirely - Honcho stores context but doesn't run background LLM analysis.

Why Two Layers?

A single memory system can't be both a curated wiki and a conversational log. The vault requires deliberate writing to be useful - that's its strength and its weakness. Honcho captures everything automatically but has no curation - every message is equal weight. Together they approximate how human memory works: some things you deliberately study and record (vault), others you just remember from experience (Honcho). Agent Tina bridges both. This is the Way.
