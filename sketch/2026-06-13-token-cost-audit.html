48% of fixed cost = three loaded skills

The research-to-web-publishing skill alone weighs 52 KB (~13,000 tokens). It carries prabha-taste (35 KB, ~8,900 tokens) and hermes-cost-optimization (10 KB, ~2,600 tokens). Together they inject ~24,500 tokens per turn — more than the entire Hermes system prompt, all tool schemas, and Honcho combined.

How 36.8K is really one turn

Fixed overhead (~50,600 tokens) + one user message (~2,000) + several tool calls (~8,000) + assistant response (~4,000) = ~64,600 tokens gross. With compression engaged at 40% threshold, contexts are summarised down to ~50% — so the post-compression view of ~36.8K is a single moderately complex turn that went past the compression threshold.

Optimization Recommendations

HIGH 1. Slim research-to-web-publishing skill

At 52 KB, this skill is the single largest token hog. It contains the entire prabha-taste design system inlined + full deployment instructions. Split it:

  • Core workflow: keep 6-8 KB (markdown→HTML steps, git push, curl verify)
  • Reference design tokens: extract into references/ directory loaded conditionally
  • Estimated savings: ~11,000 tokens per turn when design details not needed

HIGH 2. Audit and trim all loaded skills

Every skill is injected in full. For this task, 3 skills = 97 KB total. Most tasks load 8-12 skills. Key targets:

  • itsm-msp-agentic-research: 36 KB — consider splitting research templates into references/
  • html-design-systems: 11 KB — trim heavy reference tables into separate docs
  • prabha-taste: 35 KB — 95% of this is reference vocabulary never invoked in most tasks

3. Enable selective toolset loading

CLI profile loads 16 toolsets unconditionally. Most sessions only need 5-8. Configure per-task toolset trimming via enabled_toolsets in delegation or cron:

# Cron jobs can restrict loaded toolsets
cronjob(action='create', prompt='...', enabled_toolsets=['terminal', 'web'])

Estimated savings: ~2,500 tokens per turn when vision/tts/image_gen/browser are trimmed

4. Reduce Honcho verbosity

Honcho injects the full peer card + AI self-representation + all observations each turn. The data for this session alone contained ~200 observation entries. Options:

  • Reduce nudge_interval (currently 10) or increase flush_min_turns (currently 6)
  • Set memory.memory_char_limit lower (currently 2,200, USER.md is 1,375)
  • Consider dropping Honcho for CLI sessions entirely if the session_search tool covers recall needs

Estimated savings: ~1,000-2,500 tokens per turn

5. Widen smart model routing trigger

Smart model routing is enabled but the cheap model (mimo-v2.5) only triggers on queries under 160 characters / 28 words. Widen to ~400 chars / 60 words to catch more simple replies on the cheap model.

smart_model_routing:
  enabled: true
  cheap_model:
    model: mimo-v2.5
    provider: xiaomi
  max_simple_chars: 400    # was 160
  max_simple_words: 60     # was 28

6. Archive stale cron jobs

15 cron jobs exist (14 active). Each cron job's config and status is tracked. Archive or consolidate jobs that haven't produced real output in 30+ days. Especially the duplicate 6PM raw processing jobs documented in Honcho.

7. Purge stale sessions

366 sessions processed (June 8-10). Session DB auto-prune is disabled. Enable it or run a manual vacuum to reduce session_search overhead bandwidth.

Note on compression

When context exceeds 40% capacity (51,200 tokens), Hermes compresses older exchanges. Summaries replace raw messages. The 36.8K number likely represents a post-compression measurement, meaning the raw token count before compression was higher (estimated ~64,600). Compression is already saving ~28,000 tokens per overflowed turn.
