1 The Three GenUI Design Patterns

Controlled, Declarative, Open-ended. Every GenUI framework in 2026 lives somewhere on this line.

Generative UI (GenUI) is no longer a prototype concept. It is shipping in production. The last seven days saw two major milestones: Google ADK 2.0 (July 1) shipping native workflow support that blends deterministic code with agent-generated UI, and Amazon Bedrock AgentCore (June 30) adopting AG-UI as a first-class protocol, making GenUI production-ready on the largest cloud platform.

But these launches codify something more fundamental. A clear taxonomy has crystallized for how agents communicate UI to users. Every framework, from CopilotKit to MCP Apps to A2UI, lives on a spectrum defined by how much control stays in the frontend versus how much freedom the agent has.

Controlled

Frontend pre-builds every component. The agent selects which to render and fills data. High control, low freedom.

Declarative

Agent emits a structured UI schema (A2UI, Open-JSON-UI). Frontend maps schemas to components. Shared control.

Open-ended

Agent returns raw HTML/UI. Frontend sandboxes it. Low control, maximum freedom. Good for disposable UIs, risky as a primary surface.

The architectural choice is not cosmetic. Each pattern breaks your app in a different way at scale. Controlled blows up the context window with tool descriptions (400 tokens per component, 25 components = 10,000 tokens per turn before the user speaks). Declarative scales flat via a schema catalog. Open-ended introduces iframe sandbox risks.

Implication for product design: Ship Controlled for the top 3-5 pixel-perfect flows. Default to Declarative for the long tail. Reserve Open-ended for one-shot throwaway UIs. Count your render tools. Past 15, you are in Controlled and the wall is close. Wire a declarative schema this quarter.

2 Hybrid Agentic Workflows

Google ADK 2.0 (July 1) replaces pure-LLM orchestration with a graph of deterministic code nodes and bounded LLM agents.

The most significant design pattern launch this week is Google's ADK 2.0, which introduces a structured workflow runtime that separates execution routing from language processing. Instead of asking an LLM to orchestrate a multi-step business process end-to-end, developers can now compose a directed graph where deterministic nodes (tool calls, human-in-the-loop, conditional routing) run at programmatic speed, while LLM nodes handle only the ambiguous cognitive work.

The benchmark data tells the story. In a standard refund processing flow, a vanilla LLM agent consumed 5,152 tokens per run with 7.2s latency. The ADK 2.0 workflow consumed 2,265 tokens (~50% savings) at 5.7s (~20% latency reduction). The savings come from not paying the LLM to infer what comes next when B always follows A.

This is a direct response to the structural failure pattern of autonomous agents: context bloat, execution derailment, hallucinated step-skipping, and unbounded token costs. ADK 2.0's workflow graph controls data passing between nodes programmatically, preventing tool output from accumulating in the conversation history and degrading the model's focus.

The design philosophy is explicit: if you can clearly map the workflow, use determinism. LLMs are optimized for variety and creativity. Business processes require exact execution. The spectrum of control ranges from pure autonomous agents (no predetermined structure) through hybrid workflows (deterministic graph + bounded LLM nodes) to pure deterministic code (no LLM at all). Most production applications should live in the middle.

Implication for product design: Map every multi-step user flow as a directed graph before writing code. Identify which nodes genuinely need LLM reasoning (ambiguous inputs, subjective decisions) and which are deterministic (tool calls, API writes, routing). The hybrid workflow is the default production architecture. Pure autonomous agents are the exception, not the rule.

3 AG-UI Goes Production on AWS

Amazon Bedrock AgentCore (June 30) adopts AG-UI as a first-class streaming protocol, making GenUI deployable at scale with managed auth, sessions, and auto-scaling.

Before this week, deploying a GenUI agent to production meant manually configuring SSE endpoints, writing auth middleware, managing session isolation across users, and setting up auto-scaling. Amazon Bedrock AgentCore Runtime eliminates that entire category of work. It is a fully managed, framework-agnostic hosting environment that speaks AG-UI natively alongside HTTP, MCP, and A2A.

The impact on design patterns is concrete. AG-UI defines typed event categories that any frontend can consume: lifecycle events (run starts, finishes, errors), text events (token-by-token streaming), tool call events (tool firing and results), and state events (shared objects both agent and UI read/write). This standardized wire format means one frontend parser handles multiple backend frameworks (Strands, LangGraph, ADK) without changes.

CopilotKit's integration, published June 30, demonstrates a working agent on AgentCore that renders charts inline, keeps a todo canvas in sync with the agent (bidirectional), and pauses for human-in-the-loop approval. All three patterns run on the same AG-UI stream. The frontend never touches the backend directly. It consumes events.

Implication for product design: The integration boundary between agent backends and user-facing UIs has standardized. AG-UI is the wire format. If your product generates UI from agent output, you should target AG-UI as your output protocol, not a custom JSON format. This decouples frontend evolution from backend architecture changes. The frontend becomes a thin renderer of typed events.
Sources: CopilotKit: Generative UI on Bedrock AgentCore with AG-UI (Jun 30) · AWS Bedrock AgentCore AG-UI Docs · FAST Template (AWS)

4 The Four-Protocol Stack

MCP, A2A, AG-UI, and A2UI. One job each. The composable agentic UI stack is standardizing in real time.

A theme running through every launch this week is the consolidation of agent communication protocols into a composable stack. Each protocol has exactly one job, and the layering is becoming the de facto enterprise pattern:

MCP

Model Context Protocol

Connects agents to tools, data sources, and APIs. The retrieval layer.

A2A 1.0

Agent-to-Agent Protocol

Connects agents to other agents. Multi-agent orchestration and delegation.

AG-UI

Agent-User Interaction

Connects agents to users. Streams events (text, tool calls, state) over SSE. The runtime transport.

A2UI

Agent-to-User Interface

Google's declarative spec for agents emitting UI schema. Rides on AG-UI. Defines what users touch.

Oracle's approach from last month illustrates the composability best: Agent Spec + AG-UI + A2UI. Agent Spec defines what runs, AG-UI carries the interaction, and A2UI defines what the user touches. You can swap implementations at any layer while keeping the experience stable. AWS AgentCore supports MCP, A2A, and AG-UI on the same runtime. CopilotKit is the AG-UI client that decodes the stream for frontends.

Vercel's json-renderer (recent launch) supports A2UI as a proof of concept. Oracle shipped Agent Spec with A2UI + AG-UI support. The AG-UI + A2UI middleware system means any agent that speaks AG-UI can drive A2UI v0.9 on day zero with no custom integration.

Implication for product design: Design your agent's output layer to target AG-UI + A2UI, not a custom schema. Your frontend becomes a renderer of standardized events. You can swap agent backends (ADK, LangGraph, Strands, Bedrock) without touching the UI. This is the same decoupling pattern that REST APIs brought to backend services, now applied to agent-generated interfaces.

5 The GenUI Token Tax

Every registered component costs ~400 tokens in the agent's context before the user speaks. At 25 components, that is 10,000 tokens per turn. The economic shape of GenUI architecture.

Beyond the protocol stack and the pattern taxonomy, a practical economic constraint is shaping GenUI design decisions. Every UI component you register as a Controlled tool sits in the agent's context window before the user has said anything. A typical tool description with JSON schema runs ~400 tokens. Twenty-five components = 10,000 tokens on every single turn, whether the agent uses them or not.

The token tax creates a design tension. Controlled patterns offer pixel-perfect UI but scale linearly with cost. Declarative patterns (A2UI) offload the UI generation to a schema catalog, so adding a new card type requires no additional context window budget. The agent emits the schema; the frontend renders it from a single registered tool.

This is not a theoretical concern. CopilotKit's own infrastructure story is a cautionary data point. In June 2026, they swapped coding agents to a new model and ran up a $22,000 Anthropic bill in under 24 hours. The root cause was not a bug or runaway script. Five engineers doing normal work triggered it because the new model interpreted their English-prose behavioral rules more loosely. The result: bloated sub-agents, runaway review loops, exploding fanout, and oversized single turns. Their fix lived at the rules layer, swapping vague adjectives for hard numeric budgets and machine-checkable gates.

Implication for product design: The token tax is a design constraint, not just a cost concern. It dictates how many UI primitives you can offer an agent. Defaulting to Declarative (A2UI) patterns gives you unbounded UI variety at fixed context cost. Every Controlled component added to the catalog should justify its 400-token-per-turn cost. Design for token budgets the way you design for API rate limits.

Design Decision Tree: Pick on Purpose

Before writing a line of GenUI code, run the decision tree. Not picking is the most expensive choice.

if

Pixel-perfect mockups exist

Designer has exact specs for this flow? Use Controlled. Ship the top 3-5 flows this way.
else if

Dozens of card types or widgets

Catalog of UI variations with shared structure? Use Declarative (A2UI). One tool, infinite UIs.
else if

One-shot throwaway visualization

User will never see this exact view twice? Use Open-ended (MCP Apps). Sandbox it. Never as primary surface.
default

Declarative schema

Can't decide? Default to Declarative. Upgrade to Controlled for top 3 flows. Never Open-ended as default.

Cross-Cutting Observations

  • The GenUI taxonomy is now table stakes. Every major platform (Google, AWS, Oracle, Microsoft via CopilotKit) shipped or adopted GenUI protocol support this month. The question is no longer whether your app will have generative UI, but which pattern you will use.
  • The hybrid workflow is the new default architecture. Pure autonomous agents and pure deterministic code are both edge cases. Production applications will blend them in a directed graph where LLM nodes handle only ambiguous reasoning and code nodes handle everything else.
  • Protocols are the new platform moats. MCP, A2A, AG-UI, and A2UI compose into a stack that is framework-agnostic. The vendor that owns the protocol layer (not the agent framework) owns the ecosystem. Google's AG-UI adoption across AWS, Oracle, and LangChain signals that the protocol layer, not the model, is where the platform value is consolidating.
  • The token tax is a design constraint. GenUI architecture is economically bounded by context window costs. Declarative patterns win at scale because they offer unbounded UI variety at fixed per-turn cost. Products that ignore this constraint will price themselves out of production.
