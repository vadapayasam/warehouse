02

Product Ideas

IDEA 01 PSA-Native Triage and Runbook Engine HIGH CONFIDENCE

The tool lives inside the PSA ticket. When an alert fires and a ticket is created, MSPclaw surfaces an asset health snapshot pulled from the connected RMM, matched runbooks from the action-pack library, and a one-click "run on device" button inside the ticket view. The tech never leaves the PSA.

Eliminates the ConnectWise - NinjaRMM - Notion - RDP context-switch cycle Works with any PSA (not ConnectWise-only like Automate) Runbooks are community-contributed, not vendor-defined Minimum viable surface: ticket create/update/close + asset context fetch
IDEA 02 Community Action-Pack Marketplace HIGH CONFIDENCE

Copy TacticalRMM's three-tier contribution model exactly. An action pack bundles trigger conditions, runbook steps, and PSA ticket update logic together. The integration contract is action-packs.json, mirroring how TacticalRMM uses community_scripts.json. This is the proven distribution mechanism for MSP automation at scale.

WIP tier: in-progress community ideas Staging tier: community-tested, awaiting promotion Official tier: reviewed and bundled with MSPclaw installs Categories: L1 alerts, disk/CPU/memory, onboarding, offboarding, security response
IDEA 03 Alert Intelligence Layer (L1 Noise Reduction) MED CONFIDENCE

MSPs drown in alert noise. MSPclaw sits between the monitoring platform and the PSA. It deduplicates alert storms, classifies severity using community-defined explicit rules (not a black-box ML model), and auto-resolves known-good patterns. Unresolvable alerts route to the correct queue with context already attached.

Dedup: same device + same error in a time window = 1 ticket, not 40 Explicit rules, not ML: editable, auditable, trustable Auto-close known patterns: disk back under 90% - close ticket + add worklog note Context-enriched routing: queue assignment based on asset + alert type

Marketplace Model

Key Decision

AGPL v3.0 protects against SaaS reselling while maximising community adoption. MIT would grow faster but loses the managed-service moat. Tracecat proves AGPL works for technical self-hosted buyers.

License: AGPL v3.0 Verified pattern
Community WIP and staging action packs. Not bundled with installs. Anyone can contribute, no review required. FREE
Verified Officially reviewed packs. Ships bundled with every MSPclaw install. Requires a valid entry in action-packs.json. FREE
Partner Cloud Managed cloud hosting + SLAs for MSPs who don't want to self-host. Revenue comes from here, not the software. PAID HOSTING

04

Integration Priority

Minimum viable surface per PSA: ticket create/update/close + asset context fetch. That is enough to ship v1 for any connector.

# Platform Type Rationale
1 SuperOps PSA + RMM Your domain. Modern REST API. Fastest-growing. Ship first.
2 ConnectWise Manage PSA Largest PSA install base in North America. Required for credibility.
3 Halo PSA PSA Strong REST API. Fast-growing in UK/EU. Good early adopter audience.
4 Datto Autotask PSA Large install base. Older API, worth it for reach.
5 NinjaRMM RMM + PSA RMM-first, but PSA module is gaining adoption fast.
6 Atera RMM + PSA Strong SMB presence. All-in-one shops. Good for smaller MSPs.
05

Open Questions

Answer these before committing to architecture. Each one changes the build order.

1 Is there a runbook-sharing community in MSPGeek Slack or GitHub you can seed the marketplace with at launch? Without a seed library, the marketplace is empty on day one and adoption stalls.
2 AGPL vs MIT: do you want to protect against hosted resale, or maximise community adoption speed? AGPL grows slower but protects the managed-service revenue model. MIT grows fastest but any MSP tooling vendor can fork and ship.
3 What is the minimum PSA API surface for ConnectWise REST? Is it clean enough to build alongside SuperOps in v1, or does the API complexity push it to v2?
4 Is ConnectWise still dominant in the under-50-tech MSP segment, or has SuperOps/Halo taken meaningful share? This determines whether ConnectWise connector is required at launch or a fast-follow.
