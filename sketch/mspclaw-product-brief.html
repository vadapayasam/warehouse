The Orchestrator Model

Core Principle

MSPclaw never owns the data. It reads events from existing tools, executes defined playbook steps via their APIs, and writes results back. Every action is auditable. Every step is tech-approved. AI suggests but does not execute without a playbook.

Architecture: MSPclaw as orchestration layer

Every L1 scenario follows the same loop: alert fires in a source tool, MSPclaw picks it up, looks up the matching playbook, executes each step via the relevant tool's API, and writes the outcome back to the PSA. The PSA remains the single source of truth.

Why "tech controls, AI assists"

MSPs must explain to clients exactly what ran on their infrastructure. "The AI did something" is not an acceptable incident report. A versioned, auditable playbook is. AI can suggest steps when building a new playbook, but the tech reviews and approves before it ever executes in production.

Playbooks: The Core Primitive

A playbook is a deterministic, versioned sequence of steps that resolves a specific L1 scenario. Not a script. Not a workflow diagram. A defined sequence of tool actions with explicit branching logic. Techs read it, test it, trust it.

Example: disk-full-windows-server.playbook.yaml Official tier
T
Trigger: disk_used_percent > 90
Matches alert from NinjaRMM, Zabbix, or ConnectWise Automate. Extracts asset ID and disk details.
source: rmm_alert | psa_ticket_created
1
Run: get_disk_breakdown.ps1
Executes script on asset via RMM API. Returns top 10 directories by size.
via: rmm.run_script(asset_id, script)
2
Run: clear_temp_and_logs.ps1
Clears known safe paths: Windows temp, IIS logs, CBS logs. Returns bytes freed.
via: rmm.run_script(asset_id, script)
?
Branch: disk_used_percent still > 85?
Re-checks disk usage after cleanup.
if yes: step 4 (escalate) / if no: step 5 (resolve)
W
Write back: update ticket + worklog
Updates PSA ticket with steps taken, bytes freed, current disk %. Adds worklog entry. Closes or escalates to L2 queue.
via: psa.update_ticket() | psa.add_worklog()
# disk-full-windows-server.playbook.yaml id: disk-full-windows-server version: 2.1.0 tier: official psa_support: [superops, connectwise, halo, autotask] rmm_support: [ninja, tactical, automate] trigger: conditions: - alert.type == "disk" - alert.disk_percent > 90 steps: - id: get_breakdown action: rmm.run_script script: scripts/get_disk_breakdown.ps1 - id: clear_temp action: rmm.run_script script: scripts/clear_temp_and_logs.ps1 - id: check_after action: rmm.get_disk_percent branch: "> 85": escalate_l2 "<= 85": resolve - id: resolve action: psa.update_ticket status: resolved worklog: auto_generated
04

Playbook Builder Tool

Key Insight

The builder is what makes the marketplace possible. Without a low-friction way to create and test playbooks, only developers contribute. With a builder, L1 techs contribute from operational knowledge they actually have.

BUILDER TOOL
Visual Step Editor with Test Mode
NEW IDEA

A browser-based tool for building and testing playbooks without writing YAML by hand. Each step is a drag-and-drop action block. Trigger conditions use a form UI, not a query language. Test mode runs against a sandbox asset with dry-run mode enabled on all write operations. AI can suggest a full playbook draft from a natural-language description ("disk cleanup on Windows Server"), but the tech reviews every step before saving.

Drag-and-drop step sequencer: add rmm.run_script, psa.update_ticket, check_condition blocks Trigger builder: condition form with autocomplete for alert fields Test mode: dry-run against real asset, shows exact API calls before they execute AI assist: suggest full draft from text description, tech reviews and edits each step One-click publish to community staging tier, with diff view vs previous version Version history: every change tracked, rollback to previous version in one click

Product Ideas Summary

IDEA 01
Orchestration Engine (core product)
HIGH CONFIDENCE

The runtime that listens for events from connected tools, matches and executes playbooks, and writes results back to the PSA. Self-hosted, AGPL-licensed, no execution limits. Every action is logged and attributable.

Webhook receiver for all connected tools (configurable per connector) Playbook matching engine: trigger conditions evaluated against event fields Step executor: calls tool APIs in sequence, captures output Write-back: updates PSA ticket with steps taken + auto-generated worklog
IDEA 02
Community Playbook Marketplace
HIGH CONFIDENCE

Three-tier contribution model copied from TacticalRMM community-scripts (verified pattern). Each playbook is a YAML file with trigger conditions, step sequence, and PSA/RMM support matrix. Community contributes, maintainers review, verified packs ship bundled with every MSPclaw install.

WIP tier: any community member can publish a draft Staging tier: community-tested, awaiting maintainer review Official tier: reviewed, shipped bundled with MSPclaw installs Fork model: fork any community playbook, customize, publish your variant
IDEA 03
Playbook Builder Tool
NEW IDEA

Visual step editor + test mode. The tool that makes the marketplace possible. Tech-controlled: every step visible, every write operation preview-able before it runs. See section 04 for full detail.

Drag-and-drop step sequencer, no YAML required Dry-run test mode against real assets AI suggest mode for drafting new playbooks from description
IDEA 04
Alert Intelligence Layer (L1 noise reduction)
MED CONFIDENCE

Deduplication + classification layer between monitoring tools and the PSA. Same device, same error, same time window = 1 ticket instead of 40. Rules are explicit YAML, not a black-box model. MSPs can read and audit them.

Dedup window: configurable per alert type (e.g. 15 min for disk alerts) Auto-resolve: if condition clears before playbook runs, close with note Rule editor: same builder UI as playbook editor

Integration Priority

Minimum viable surface per connector: receive webhook events + run scripts on assets (RMM) + create/update/close tickets + add worklogs (PSA).

# Platform Role Rationale
1 SuperOps PSA + RMM Your domain. Modern REST API. Clean webhook support. Ship first.
2 ConnectWise Manage PSA Largest PSA install base in North America. Webhook support is flaky - poll as fallback.
3 NinjaRMM RMM Primary RMM for executing playbook scripts. Good REST API for script execution.
4 Halo PSA PSA Clean API. Fast-growing in UK/EU. Good early-adopter community.
5 Datto Autotask PSA Large install base. Older API but worth it for reach.
6 Atera RMM + PSA All-in-one for smaller MSPs. Single connector covers both layers.

Open Questions

1 Is there a runbook or playbook-sharing community in MSPGeek Slack or GitHub that could seed the marketplace at launch? Without seed content, the marketplace is empty on day one.
2 AGPL vs MIT: protect against hosted resale (AGPL) or maximise adoption speed (MIT)? AGPL grows slower but protects the managed-service revenue angle.
3 What should the builder tool ship as? Web app bundled with MSPclaw, or a standalone hosted tool that pushes playbooks to your self-hosted instance via API?
4 ConnectWise webhook reliability: is polling acceptable for v1, or should the CW connector be deprioritised in favour of a cleaner SuperOps + NinjaRMM + Halo v1 set?
5 How does the playbook test mode access real assets safely? Sandbox RMM environment, a dedicated test asset per customer, or opt-in dry-run mode per script step?
