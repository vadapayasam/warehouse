What Doesn't Change

Heavy reasoning — code review across 10 files, research synthesis, complex refactors — stays on DeepSeek V4 Flash. The split isn't "all local or all cloud." It's grunt work local, thinking work cloud. The frontier does what it's good at (deep reasoning); the local does what it's good at (fast, cheap, structured).

The Failed Experiments

Getting local structured output working required failing three times first. Each failure taught us something specific about how 4-bit models behave.

Attempt 1
Plain-Text JSON Prompts · 40% Pass Rate

Prompted the model with "respond with JSON: {\"pass\": true}" as plain text. The 4-bit quantization corrupts structured output: random tokens slip in, closing braces get dropped, keys get hallucinated. Out of 10 attempts, only 4 produced valid JSON.

FAILED
# What we did wrong:
response = client.chat.completions.create(
    model="qwen3.5-4b-4bit",
    messages=[{"role": "user",
      "content": "Return JSON: {\"pass\":true,\"score\":0.95}"}]
)
# What we got (60% of the time):
'{"pass":true,"sc'  # truncated at the 4-bit noise point
Attempt 2
No --no-thinking Flag · 50-72% Token Waste

Without the flag, RapidMLX generates reasoning tokens before every answer. For a 100-token JSON response, the model first generates 200-400 tokens of internal monologue. That's not just wasted time, it's wasted memory context, and the reasoning output pollutes the tool-calling parser.

FAILED
# Server started WITHOUT --no-thinking:
rapid-mlx serve --model qwen3.5-4b-4bit  # 50%+ tokens = reasoning

# A simple JSON gate took 7s + 320 reasoning tokens
# before producing a 45-token answer.
Attempt 3
Non-API Tool Calling · Broken Hermes Parser

RapidMLX has a custom Hermes parser that expects OpenAI-compatible tool-calling format. Sending plain messages or using the wrong parameter format means the parser doesn't engage, and we get raw model output at 4-bit, which is unreliable for structured data.

FAILED
# Wrong approach:
curl http://127.0.0.1:8765/v1/chat/completions \
  -d '{"model":"qwen3.5-4b-4bit","messages":[...]}'

# The Hermes parser needs the 'tools' parameter to activate.
# Without it, the model speaks free-form and 4-bit noise creeps in.
Attempt 4 · Working
Tools API + --no-thinking + Persistent Server · 100% Pass Rate

The combination: proper OpenAI tools parameter, --no-thinking flag, and a persistent server that keeps the model in memory. The Hermes parser handles auto-recovery of corrupted 4-bit tokens. Result: 10/10 JSON gates pass.

WORKING
# Three things had to be right simultaneously:
rapid-mlx serve --model qwen3.5-4b-4bit --no-thinking

# Client sends proper tools API:
response = client.chat.completions.create(
    model="qwen3.5-4b-4bit",
    tools=[{"type": "function", "function": {
        "name": "gate_result",
        "parameters": {"type": "object", "properties": {
            "pass": {"type": "boolean"},
            "reason": {"type": "string"}
        }}
    }}],
    tool_choice={"type": "function",
                 "function": {"name": "gate_result"}}
)

Infrastructure

  • Server: rapid-mlx serve --model qwen3.5-4b-4bit --no-thinking --port 8765
  • Watchdog: Cron job rapid-mlx-server at */5 * * * *, no_agent=True, checks GET /v1/models health endpoint
  • Script: ~/.hermes/scripts/rapid-mlx-server.sh — starts server, validates health, restarts on failure
  • Startup time: ~9s cold, ~0.08s cached (prompt cache builds with use)
  • Fallback: King Coder3B MCP (mlx_codegen, mlx_summarize, etc.) remains independent — 0MB idle, on-demand only

Key Principle: Two Independent Inference Lines

RapidMLX and King Coder3B are separate, independent paths. If the persistent server goes down, King Coder3B picks up. If King Coder3B breaks, RapidMLX keeps running. Redundancy, not replacement. The cron watchdog keeps RapidMLX alive automatically.

Working Examples

JSON Gate (100% reliable)

from openai import OpenAI
import json

client = OpenAI(
    base_url="http://127.0.0.1:8765/v1",
    api_key="not-needed"
)

def validate_json(data: dict) -> dict:
    """Gate: is this valid JSON? Returns pass/fail + reason."""
    response = client.chat.completions.create(
        model="qwen3.5-4b-4bit",
        messages=[{
            "role": "user",
            "content": f"Validate this JSON: {json.dumps(data)}"
        }],
        tools=[{"type": "function", "function": {
            "name": "gate_result",
            "description": "JSON validation result",
            "parameters": {
                "type": "object",
                "properties": {
                    "pass": {
                        "type": "boolean",
                        "description": "Is the JSON valid?"
                    },
                    "score": {
                        "type": "number",
                        "description": "Confidence 0.0-1.0"
                    },
                    "issues": {
                        "type": "array",
                        "items": {"type": "string"},
                        "description": "Any issues found"
                    }
                },
                "required": ["pass", "score"]
            }
        }}],
        tool_choice={"type": "function",
                     "function": {"name": "gate_result"}},
        temperature=0.1
    )
    args = json.loads(
        response.choices[0].message.tool_calls[0].function.arguments
    )
    return args

# Usage
result = validate_json({"name": "Tina", "role": "agent"})
print(result)  # {'pass': True, 'score': 1.0, 'issues': []}

    """Extract errors, warnings, and key stats from a build log."""
    response = client.chat.completions.create(
        model="qwen3.5-4b-4bit",
        messages=[{

            "content": f"Summarize this Xcode build log:\n\n{log_text[:3000]}"
        }],
        tools=[{"type": "function", "function": {
            "name": "xcode_summary",
            "parameters": {
                "type": "object",
                "properties": {
                    "build_result": {"type": "string",
                        "enum": ["SUCCEEDED", "FAILED"]},
                    "error_count": {"type": "integer"},
                    "warning_count": {"type": "integer"},
                    "key_errors": {"type": "array",
                        "items": {"type": "string"}},
                    "duration_sec": {"type": "number"}
                },
                "required": ["build_result", "error_count"]
            }
        }}],
        tool_choice={"type": "function",
                     "function": {"name": "xcode_summary"}},
        temperature=0.1
    )
    return json.loads(
        response.choices[0].message.tool_calls[0].function.arguments
    )

macOS Helper Generator

def gen_macos_helper(task_desc: str) -> str:
    """Generate a macOS shell helper for a given task."""
    response = client.chat.completions.create(
        model="qwen3.5-4b-4bit",
        messages=[{
            "role": "user",
            "content": f"Write a macOS zsh helper: {task_desc}"
        }],
        max_tokens=800,
        temperature=0.2
    )
    return response.choices[0].message.content

When Idle

The model stays loaded in RAM (~1.7GB). No CPU usage, minimal memory pressure. Mac Mini M2 has 16GB total, so this is ~11% of available RAM. The cron watchdog does a simple HTTP health check (no LLM, no token cost).

Lessons Learned

1. 4-bit Models Don't Do Raw Structured Output

The quantization introduces token-level noise. Asking a 4-bit model to produce JSON without a template is like asking someone with a stutter to do radio voice-over. The tools API provides the template that compensates for the noise.

2. --no-thinking Is Not Optional

Without it, 50-72% of output tokens are reasoning — internal monologue that adds zero value for tool-calling tasks. The Hermes parser also works better when it doesn't have to separate reasoning from action.

3. Persistent Server > On-Demand for Repeated Work

The 0.08s vs 8s difference compounds. A JSON gate that runs 20 times/day saves 158 seconds of cold-start time. Over a week, that's 18 minutes. Over a month, 79 minutes. The 1.7GB RAM is the cost of that speed. For JSON gate workloads, the tradeoff is worth it.

4. Redundancy Over Single-Point Architecture

Two independent inference lines (RapidMLX server + King Coder3B MCP) means no single failure takes down local inference. The cron watchdog automates recovery. This cost (2.9GB + 1.6GB disk) against 46GB free — acceptable.

5. The Frontier Is Still Needed

Local 3-4B models handle structure and speed. But deep reasoning, multi-file code review, research synthesis, and creative work still need frontier models. The split is: grunt = local, think = cloud.
