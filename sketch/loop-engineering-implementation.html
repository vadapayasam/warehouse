What is a Loop?

A loop is a repeating cycle: Act → Observe (feedback from the environment) → Decide → Act → ... until a termination condition is met. This is the ReAct pattern (Reason + Act) that underpins every modern agent architecture. Unlike a static chain (A → B → C), a loop adapts based on feedback from the environment - compiler output, test failures, API responses, ticket state.

The key shift: you don't write prompts anymore. You write the system that writes the prompts. Your job moves from being the human-in-the-loop to being the loop architect. You define the shape of the work, the conditions that trigger it, the validation gates, and what happens when something fails. The agent executes. You design.

# The abstract loop lifecycle
WHILE termination_condition NOT met:
  1. SELECT task from queue
  2. ASSIGN to sub-agent with context + skills
  3. EXECUTE (agent acts on environment)
  4. OBSERVE (compiler, tests, API, human review)
  5. DECIDE (pass, fail, retry, escalate)
  6. UPDATE state (memory / ticket / file)
# The loop IS the program. Each agent call is just one instruction.

The 6 Pillars of a Loop

Every loop needs five core components plus one external memory layer. These are the building blocks - present in Codex, Claude Code, and Hermes.

Pillar 1

Automations

Schedule-driven triggers that perform discovery, triage, and recurring work without human initiation. The heartbeat of the loop.

Pillar 2

Worktrees

Isolated working directories so parallel agents don't collide. Each agent gets its own branch + checkout via git worktree.

Pillar 3

Skills

Codified project knowledge - SKILL.md files with conventions, build steps, constraints. Stops the agent from re-deriving everything from scratch.

Pillar 4

Plugins & Connectors

MCP-based integrations that let the loop touch real tools - issue trackers, Slack, databases, APIs, deploy systems.

Pillar 5

Sub-agents

Split the maker from the checker. One agent explores/drafts, another reviews/verifies. Never let the code writer grade its own homework.

Pillar 6

Memory / State

Persistent external state - markdown file, Linear ticket, SQLite row - that survives between agent conversations. The agent forgets; the repo doesn't.

Assessment

All 6 pillars are already present and running in our Hermes stack. The gap is not infrastructure - it's orchestration. Our cron jobs and tools run independently; they don't chain into each other with a feedback loop. The work ahead is wiring them together: cron job A's output becomes sub-agent B's context, whose result updates state C, which triggers cron job D's next tick with modified parameters.

Designing True Loops: From Pipeline to Feedback System

Our current cron jobs are pipelines, not loops. A pipeline runs A → B → C and stops. A loop runs A → B → C → CHECK → IF PASS: stop / IF FAIL: retry with modified parameters. The feedback signal is the critical missing piece.

Pattern 1: Research + Review Loop

The most impactful loop to build first - our existing research cron jobs run without verification. A verifier sub-agent turns a pipeline into a loop.

# Research Loop - Hermes-native implementation

cronjob(interval='daily', prompt='''
1. EXTRACT: web_search + web_extract for topic X
2. DRAFT: synthesize findings into markdown
3. VERIFY: delegate_task spawns a verifier sub-agent
   Verifier instructions: "Check for factual errors,
   missing sources, contradictory claims. Return
   PASS or FAIL with specific issues."
4. DECIDE based on verifier output:
   PASS → convert to HTML, publish, update state
   FAIL → fix flagged issues, re-verify (max 2 retries)
   ESCALATE → write to triage file, notify user
5. UPDATE state file: topic, date, result, retry count
''')

# Key: The verifier sub-agent is a DIFFERENT model instance,
# with DIFFERENT instructions. Never let the writer grade itself.

Pattern 2: Build + Test + Fix Loop

The classic software engineering loop - but automated. Uses terminal for build feedback as the environmental signal.

# Build-Validate Loop

1. TRIGGER: git push detected (or cron on schedule)
2. BUILD: xcodebuild or make
3. OBSERVE: capture compiler output, test results
4. DECIDE:
   PASS → update state, notify, done
   FAIL → delegate to fix sub-agent with error context
      Fix agent has access to: project skills, error logs,
      git diff, recent commit history
      Applies fix → re-build → re-check
      Max 3 retries before human escalation
5. MEMORY: write fix attempt + result to state file

# Token cost management: only expensive model on VERIFY step
# Build/compile uses mlx_coder (local, 0 token cost)

Pattern 3: Daily Briefing with Memory Carryover

A loop that remembers what happened yesterday and can reference it without re-reading history.

# Carryover Loop via Honcho + state file

1. cron triggers daily briefing generation
2. LOAD prior state: honcho_search('yesterday briefing state')
   + read last briefing's state file from sketch/
3. GATHER new data: git log, session search, vault updates
4. SYNTHESIZE: what changed since yesterday?
   + what was mentioned yesterday that needs follow-up?
5. GENERATE briefing HTML with delta annotations:
   - NEW items highlighted with since yesterday
   - Follow-up items with FOLLOW-UP
   - Closed items crossed out
6. WRITE state to honcho: 'briefing YYYY-MM-DD complete'
7. If follow-up items exist, delegate to research sub-agent

# Memory is the spine - without it, every run starts at zero

Token Budget Strategy

Loop engineering changes token economics. Our cost profile is uniquely favorable because we already distribute workloads across cost tiers:

Workload Model Token Cost Used For
Loop orchestration deepseek-v4-flash (opencode-go) $0 (flat monthly) Cron trigger, task routing, memory update, decision logic
Code generation mlx_coder (Qwen3.5-4B local) $0 (local MLX) Fix scripts, code patches, Xcode build fixes
Vision analysis mlx_vision (Qwen2-VL-2B local) $0 (local MLX) Screenshot checks, UI validation, design review
Semantic reasoning Honcho dialectic LLM $0 (self-hosted) Memory synthesis, profile summarization, context retrieval
Heavy lifting Frontier (deepseek-v4, gpt-5, etc.) Per-token (used sparingly) Complex review, architectural decisions, contentious verifications

Key insight: The loop orchestration layer (cron + routing + decision logic + memory) runs entirely on flat-rate and local models. The per-token frontier models are reserved for the verification sub-agent - the most critical but lowest-volume step. This makes the loop economically viable even at high frequency. A 15-min research loop costs essentially $0 in token overhead because the heavy model only fires on the verify step.

Next Steps: What to Build in Order

We already have the infrastructure. Here is what to wire up next, prioritized by impact and dependency order:

  1. 01
    Add a verifier sub-agent to the tweet generator cron
    Split the current single-agent cron into: writer (generates content) + verifier (checks for banned starts, char limits, dedup against yesterday, factual accuracy). If verifier fails, regenerate with error feedback. Blocks publishing until verifier PASS.
  2. 02
    State file per loop
    Add a _state.json file in sketch/ for each loop cron. Structure: {"last_run": "ISO", "last_topics": [...], "failures": [...], "retry_count": 0, "next_pending": "..."}. The cron job reads and updates this every tick. This is the memory that makes the loop continuous rather than stateless.
  3. 03
    Research loop with automated verification
    Wire the skills-radar and other research crons through a verifier sub-agent before HTML generation. Research quality improves by 30-50% with automated fact-checking before publish, and it prevents the "published bad data" failure mode.
  4. 04
    Build-fix loop for Xcode projects
    When CI fails or a local build breaks: terminal(capture error) → delegate_task(send to mlx_coder with project skills + error) → apply fix → rebuild → check → PR if passes. This is the highest-ROI loop for iOS work.
  5. 05
    Cross-loop coordination via central state
    When multiple loops exist, route their state through a single Honcho space so the monthly synthesis cron can reference daily tweet topics, the build-fix loop's failure patterns, and the research loop's discoveries in one briefing.
  6. 06
    Dashboard / feedback surface
    A simple HTML page that consumes the state files and shows: what each loop did today, pass/fail counts, retry rates, pending items. A human-accessible view of the autonomous system's health.
